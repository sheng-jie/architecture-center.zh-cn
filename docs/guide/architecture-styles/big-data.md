<span data-ttu-id="022a4-110">大多数大数据体系结构都包括下列组件中的一些或全部：</span><span class="sxs-lookup"><span data-stu-id="022a4-110">Most big data architectures include some or all of the following components:</span></span>

- <span data-ttu-id="022a4-111">**数据源**：所有大数据解决方案都以一个或多个数据源开始。</span><span class="sxs-lookup"><span data-stu-id="022a4-111">**Data sources**: All big data solutions start with one or more data sources.</span></span> <span data-ttu-id="022a4-112">示例包括：</span><span class="sxs-lookup"><span data-stu-id="022a4-112">Examples include:</span></span>

- <span data-ttu-id="022a4-116">**数据存储**：用于批处理操作的数据通常存储在分布式文件存储中，该存储可以容纳大量各种格式的大型文件。</span><span class="sxs-lookup"><span data-stu-id="022a4-116">**Data storage**: Data for batch processing operations is typically stored in a distributed file store that can hold high volumes of large files in various formats.</span></span> <span data-ttu-id="022a4-117">这类存储通常称为 *Data Lake*。</span><span class="sxs-lookup"><span data-stu-id="022a4-117">This kind of store is often called a *data lake*.</span></span> <span data-ttu-id="022a4-118">用于实现此存储的选项包括 Azure Data Lake Store 和 Azure 存储中的 blob 容器。</span><span class="sxs-lookup"><span data-stu-id="022a4-118">Options for implementing this storage include Azure Data Lake Store or blob containers in Azure Storage.</span></span> 

- <span data-ttu-id="022a4-119">**批处理**：因为数据集很大，所以，大数据解决方案通常必须使用长时间运行的批处理作业来处理数据文件，从而筛选、聚合以及准备用于分析的数据。</span><span class="sxs-lookup"><span data-stu-id="022a4-119">**Batch processing**: Because the data sets are so large, often a big data solution must process data files using long-running batch jobs to filter, aggregate, and otherwise prepare the data for analysis.</span></span> <span data-ttu-id="022a4-120">这些作业通常涉及读取源文件、对它们进行处理，以及将输出写入到新文件。</span><span class="sxs-lookup"><span data-stu-id="022a4-120">Usually these jobs involve reading source files, processing them, and writing the output to new files.</span></span> <span data-ttu-id="022a4-121">选项包括在 Azure Data Lake Analytics 中运行 U-SQL 作业，在 HDInsight Hadoop 群集中使用 Hive、Pig 或自定义 Map/Reduce 作业，或者在 HDInsight Spark 群集中使用 Java、Scala 或 Python 程序。</span><span class="sxs-lookup"><span data-stu-id="022a4-121">Options include running U-SQL jobs in Azure Data Lake Analytics, using Hive, Pig, or custom Map/Reduce jobs in an HDInsight Hadoop cluster, or using Java, Scala, or Python programs in an HDInsight Spark cluster.</span></span>

- <span data-ttu-id="022a4-122">**实时消息引入**：如果解决方案包括实时源，则体系结构必须包括一种方法来捕获并存储实时消息以进行流式处理。</span><span class="sxs-lookup"><span data-stu-id="022a4-122">**Real-time message ingestion**: If the solution includes real-time sources, the architecture must include a way to capture and store real-time messages for stream processing.</span></span> <span data-ttu-id="022a4-123">这可以是一个简单的数据存储，将在其中将传入消息放置在一个文件夹中以进行处理。</span><span class="sxs-lookup"><span data-stu-id="022a4-123">This might be a simple data store, where incoming messages are dropped into a folder for processing.</span></span> <span data-ttu-id="022a4-124">不过，许多解决方案都需要一个消息引入存储来充当消息缓冲区，以及支持横向扩展处理、可靠传递和其他消息队列语义。</span><span class="sxs-lookup"><span data-stu-id="022a4-124">However, many solutions need a message ingestion store to act as a buffer for messages, and to support scale-out processing, reliable delivery, and other message queuing semantics.</span></span> <span data-ttu-id="022a4-125">选项包括 Azure 事件中心、Azure IoT 中心和 Kafka。</span><span class="sxs-lookup"><span data-stu-id="022a4-125">Options include Azure Event Hubs, Azure IoT Hubs, and Kafka.</span></span>

- <span data-ttu-id="022a4-126">**流式处理**：捕获实时消息后，解决方案必须通过筛选、聚合以及准备用于分析的数据来处理消息。</span><span class="sxs-lookup"><span data-stu-id="022a4-126">**Stream processing**: After capturing real-time messages, the solution must process them by filtering, aggregating, and otherwise preparing the data for analysis.</span></span> <span data-ttu-id="022a4-127">然后，会将处理后的流数据写入到输出接收器。</span><span class="sxs-lookup"><span data-stu-id="022a4-127">The processed stream data is then written to an output sink.</span></span> <span data-ttu-id="022a4-128">Azure 流分析基于不断运行的 SQL 查询提供托管流处理服务，这些查询对无限的流进行操作。</span><span class="sxs-lookup"><span data-stu-id="022a4-128">Azure Stream Analytics provides a managed stream processing service based on perpetually running SQL queries that operate on unbounded streams.</span></span> <span data-ttu-id="022a4-129">还可以在 HDInsight 群集中使用开源 Apache 流式处理技术，例如 Storm 和 Spark 流式处理。</span><span class="sxs-lookup"><span data-stu-id="022a4-129">You can also use open source Apache streaming technologies like Storm and Spark Streaming in an HDInsight cluster.</span></span>

- <span data-ttu-id="022a4-130">**分析数据存储**：许多大数据解决方案会准备用于分析的数据，然后以结构化格式提供已处理的数据，可以使用分析工具查询这些数据。</span><span class="sxs-lookup"><span data-stu-id="022a4-130">**Analytical data store**: Many big data solutions prepare data for analysis and then serve the processed data in a structured format that can be queried using analytical tools.</span></span> <span data-ttu-id="022a4-131">如大多数传统业务智能 (BI) 解决方案中所见，用来为这些查询提供服务的分析数据存储可以是 Kimball 样式的关系数据仓库。</span><span class="sxs-lookup"><span data-stu-id="022a4-131">The analytical data store used to serve these queries can be a Kimball-style relational data warehouse, as seen in most traditional business intelligence (BI) solutions.</span></span> <span data-ttu-id="022a4-132">另外，可以通过低延迟 NoSQL 技术（例如 HBase，或通过分布式数据存储中的数据文件提供元数据抽象的交互式 Hive 数据库）来提供数据。</span><span class="sxs-lookup"><span data-stu-id="022a4-132">Alternatively, the data could be presented through a low-latency NoSQL technology such as HBase, or an interactive Hive database that provides a metadata abstraction over data files in the distributed data store.</span></span> <span data-ttu-id="022a4-133">Azure SQL 数据仓库针对大规模的基于云的数据仓库提供托管服务。</span><span class="sxs-lookup"><span data-stu-id="022a4-133">Azure SQL Data Warehouse provides a managed service for large-scale, cloud-based data warehousing.</span></span> <span data-ttu-id="022a4-134">HDInsight 支持交互式 Hive、HBase 和 Spark SQL，也可以使用这些技术来提供用于分析的数据。</span><span class="sxs-lookup"><span data-stu-id="022a4-134">HDInsight supports Interactive Hive, HBase, and Spark SQL, which can also be used to serve data for analysis.</span></span>

## <a name="best-practices"></a><span data-ttu-id="022a4-180">最佳做法</span><span class="sxs-lookup"><span data-stu-id="022a4-180">Best practices</span></span>

- <span data-ttu-id="022a4-181">**利用并行**。</span><span class="sxs-lookup"><span data-stu-id="022a4-181">**Leverage parallelism**.</span></span> <span data-ttu-id="022a4-182">大多数大数据处理技术会将工作负荷分布到多个处理单元中。</span><span class="sxs-lookup"><span data-stu-id="022a4-182">Most big data processing technologies distribute the workload across multiple processing units.</span></span> <span data-ttu-id="022a4-183">这需要以可拆分格式创建和存储静态数据文件。</span><span class="sxs-lookup"><span data-stu-id="022a4-183">This requires that static data files are created and stored in a splittable format.</span></span> <span data-ttu-id="022a4-184">分布式文件系统（例如 HDFS）可以优化读取和写入性能，并且通过多个群集节点并行执行实际处理，这降低了整体作业时间。</span><span class="sxs-lookup"><span data-stu-id="022a4-184">Distributed file systems such as HDFS can optimize read and write performance, and the actual processing is performed by multiple cluster nodes in parallel, which reduces overall job times.</span></span>

- <span data-ttu-id="022a4-185">**将数据分区**。</span><span class="sxs-lookup"><span data-stu-id="022a4-185">**Partition data**.</span></span> <span data-ttu-id="022a4-186">批处理通常按定期计划进行 &mdash; 例如，每周或每月。</span><span class="sxs-lookup"><span data-stu-id="022a4-186">Batch processing usually happens on a recurring schedule &mdash; for example, weekly or monthly.</span></span> <span data-ttu-id="022a4-187">请根据与处理计划匹配的时间期间对数据文件和数据结构（例如表）进行分区。</span><span class="sxs-lookup"><span data-stu-id="022a4-187">Partition data files, and data structures such as tables, based on temporal periods that match the processing schedule.</span></span> <span data-ttu-id="022a4-188">这可以简化数据引入和作业计划，并且能够更轻松地排除故障。</span><span class="sxs-lookup"><span data-stu-id="022a4-188">That simplifies data ingestion and job scheduling, and makes it easier to troubleshoot failures.</span></span> <span data-ttu-id="022a4-189">另外，对在 Hive、U-SQL 或 SQL 查询中使用的表进行分区可以显著提高查询性能。</span><span class="sxs-lookup"><span data-stu-id="022a4-189">Also, partitioning tables that are used in Hive, U-SQL, or SQL queries can significantly improve query performance.</span></span>

- <span data-ttu-id="022a4-190">**应用读取时架构语义**。</span><span class="sxs-lookup"><span data-stu-id="022a4-190">**Apply schema-on-read semantics**.</span></span> <span data-ttu-id="022a4-191">使用 Data Lake，可以组合使用多种格式的文件存储，无论是结构化的、半结构化的还是非结构化的。</span><span class="sxs-lookup"><span data-stu-id="022a4-191">Using a data lake lets you to combine storage for files in multiple formats, whether structured, semi-structured, or unstructured.</span></span> <span data-ttu-id="022a4-192">使用*读取时架构*语义，这些语义在处理数据时而非存储数据时将架构投射到数据上。</span><span class="sxs-lookup"><span data-stu-id="022a4-192">Use *schema-on-read* semantics, which project a schema onto the data when the data is processing, not when the data is stored.</span></span> <span data-ttu-id="022a4-193">这增加了解决方案的灵活性，并且可以防止在数据引入期间由于数据验证和类型检查而导致瓶颈。</span><span class="sxs-lookup"><span data-stu-id="022a4-193">This builds flexibility into the solution, and prevents bottlenecks during data ingestion caused by data validation and type checking.</span></span>

- <span data-ttu-id="022a4-194">**就地处理数据**。</span><span class="sxs-lookup"><span data-stu-id="022a4-194">**Process data in-place**.</span></span> <span data-ttu-id="022a4-195">传统 BI 解决方案通常使用提取、转换和加载 (ETL) 流程将数据移动到数据仓库。</span><span class="sxs-lookup"><span data-stu-id="022a4-195">Traditional BI solutions often use an extract, transform, and load (ETL) process to move data into a data warehouse.</span></span> <span data-ttu-id="022a4-196">大数据解决方案用于处理更大的数据量和更多样化的格式，它们通常使用 ETL 的变体，例如转换、提取和加载 (TEL)。</span><span class="sxs-lookup"><span data-stu-id="022a4-196">With larger volumes data, and a greater variety of formats, big data solutions generally use variations of ETL, such as transform, extract, and load (TEL).</span></span> <span data-ttu-id="022a4-197">使用此方法时，将在分布式数据存储中处理数据，将其转换为所需的结构，然后将转换后的数据移动到分析数据存储中。</span><span class="sxs-lookup"><span data-stu-id="022a4-197">With this approach, the data is processed within the distributed data store, transforming it to the required structure, before moving the transformed data into an analytical data store.</span></span>

- <span data-ttu-id="022a4-198">**平衡利用率和时间成本**。</span><span class="sxs-lookup"><span data-stu-id="022a4-198">**Balance utilization and time costs**.</span></span> <span data-ttu-id="022a4-199">对于批处理作业，请务必考虑两个因素：计算节点的每单位成本，以及使用这些节点完成作业的每分钟成本。</span><span class="sxs-lookup"><span data-stu-id="022a4-199">For batch processing jobs, it's important to consider two factors: The per-unit cost of the compute nodes, and the per-minute cost of using those nodes to complete the job.</span></span> <span data-ttu-id="022a4-200">例如，某个批处理作业可能需要四个群集节点并花费八个小时。</span><span class="sxs-lookup"><span data-stu-id="022a4-200">For example, a batch job may take eight hours with four cluster nodes.</span></span> <span data-ttu-id="022a4-201">但是，该作业可能只是在前两个小时内使用所有四个节点，在那之后，可能只需要两个节点。</span><span class="sxs-lookup"><span data-stu-id="022a4-201">However, it might turn out that the job uses all four nodes only during the first two hours, and after that, only two nodes are required.</span></span> <span data-ttu-id="022a4-202">在这种情况下，在两个节点上运行整个作业将增加总的作业时间，但是不会使其翻倍，因此总成本更低。</span><span class="sxs-lookup"><span data-stu-id="022a4-202">In that case, running the entire job on two nodes would increase the total job time, but would not double it, so the total cost would be less.</span></span> <span data-ttu-id="022a4-203">在某些业务方案中，与使用利用率低下的群集资源而导致更高的成本相比，延长处理时间可能更为合适。</span><span class="sxs-lookup"><span data-stu-id="022a4-203">In some business scenarios, a longer processing time may be preferable to the higher cost of using under-utilized cluster resources.</span></span>

- <span data-ttu-id="022a4-204">**单独的群集资源**。</span><span class="sxs-lookup"><span data-stu-id="022a4-204">**Separate cluster resources**.</span></span> <span data-ttu-id="022a4-205">在部署 HDInsight 群集时，通常会通过为每种类型的工作负荷预配单独的群集资源来获得更好的性能。</span><span class="sxs-lookup"><span data-stu-id="022a4-205">When deploying HDInsight clusters, you will normally achieve better performance by provisioning separate cluster resources for each type of workload.</span></span> <span data-ttu-id="022a4-206">例如，尽管 Spark 群集包括 Hive，但是，如果需要通过 Hive 和 Spark 执行大量处理，则应当考虑部署单独的专用 Spark 和 Hadoop 群集。</span><span class="sxs-lookup"><span data-stu-id="022a4-206">For example, although Spark clusters include Hive, if you need to perform extensive processing with both Hive and Spark, you should consider deploying separate dedicated Spark and Hadoop clusters.</span></span> <span data-ttu-id="022a4-207">类似地，如果使用 HBase 和 Storm 进行低延迟流处理并使用 Hive 进行批处理，请考虑为 Storm、HBase 和 Hadoop 部署单独的群集。</span><span class="sxs-lookup"><span data-stu-id="022a4-207">Similarly, if you are using HBase and Storm for low latency stream processing and Hive for batch processing, consider separate clusters for Storm, HBase, and Hadoop.</span></span>

- <span data-ttu-id="022a4-208">**安排数据引入**。</span><span class="sxs-lookup"><span data-stu-id="022a4-208">**Orchestrate data ingestion**.</span></span> <span data-ttu-id="022a4-209">在某些情况下，现有业务应用程序可以将用于批处理的数据文件直接写入到 Azure 存储 Blob 容器中，HDInsight 或 Azure Data Lake Analytics 可以从该容器中使用这些数据文件。</span><span class="sxs-lookup"><span data-stu-id="022a4-209">In some cases, existing business applications may write data files for batch processing directly into Azure storage blob containers, where they can be consumed by HDInsight or Azure Data Lake Analytics.</span></span> <span data-ttu-id="022a4-210">不过，你通常需要安排将数据从内部或外部数据源引入到 Data Lake 中。</span><span class="sxs-lookup"><span data-stu-id="022a4-210">However, you will often need to orchestrate the ingestion of data from on-premises or external data sources into the data lake.</span></span> <span data-ttu-id="022a4-211">可以使用某个业务流程工作流或管道（例如 Azure 数据工厂或 Oozie 支持的那些）以可预测且可集中管理的方式实现此目的。</span><span class="sxs-lookup"><span data-stu-id="022a4-211">Use an orchestration workflow or pipeline, such as those supported by Azure Data Factory or Oozie, to achieve this in a predictable and centrally manageable fashion.</span></span>
