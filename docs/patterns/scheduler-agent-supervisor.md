# <a name="scheduler-agent-supervisor-pattern"></a><span data-ttu-id="dca5d-104">计划程序代理监督程序模式</span><span class="sxs-lookup"><span data-stu-id="dca5d-104">Scheduler Agent Supervisor pattern</span></span>

[!INCLUDE [header](../_includes/header.md)]

<span data-ttu-id="dca5d-105">以单项操作的形式协调一组分布式操作。</span><span class="sxs-lookup"><span data-stu-id="dca5d-105">Coordinate a set of distributed actions as a single operation.</span></span> <span data-ttu-id="dca5d-106">如果任一操作失败，将尝试以透明方式处理故障，如果故障处理失败，则撤消已执行的工作，从而使整个操作作为一个整体成功或失败。</span><span class="sxs-lookup"><span data-stu-id="dca5d-106">If any of the actions fail, try to handle the failures transparently, or else undo the work that was performed, so the entire operation succeeds or fails as a whole.</span></span> <span data-ttu-id="dca5d-107">这使分布式系统可以恢复并重试因暂时性异常、长期故障和进程故障而失败的操作，从而提高自身的复原能力。</span><span class="sxs-lookup"><span data-stu-id="dca5d-107">This can add resiliency to a distributed system, by enabling it to recover and retry actions that fail due to transient exceptions, long-lasting faults, and process failures.</span></span>

## <a name="context-and-problem"></a><span data-ttu-id="dca5d-108">上下文和问题</span><span class="sxs-lookup"><span data-stu-id="dca5d-108">Context and problem</span></span>

<span data-ttu-id="dca5d-109">应用程序执行包含多个步骤的任务，其中有些步骤可能调用远程服务或访问远程资源。</span><span class="sxs-lookup"><span data-stu-id="dca5d-109">An application performs tasks that include a number of steps, some of which might invoke remote services or access remote resources.</span></span> <span data-ttu-id="dca5d-110">各个步骤可以彼此独立，但它们由实施任务的应用程序逻辑进行协调。</span><span class="sxs-lookup"><span data-stu-id="dca5d-110">The individual steps might be independent of each other, but they are orchestrated by the application logic that implements the task.</span></span>

<span data-ttu-id="dca5d-111">应用程序应尽可能确保任务完成运行，并解决访问远程服务或资源时可能发生的任何故障。</span><span class="sxs-lookup"><span data-stu-id="dca5d-111">Whenever possible, the application should ensure that the task runs to completion and resolve any failures that might occur when accessing remote services or resources.</span></span> <span data-ttu-id="dca5d-112">导致故障的原因有很多。</span><span class="sxs-lookup"><span data-stu-id="dca5d-112">Failures can occur for many reasons.</span></span> <span data-ttu-id="dca5d-113">例如，网络可能关闭，通信可能中断，远程服务可能停止响应或处于不稳定状态，或者可能因为资源约束而暂时无法访问某个远程资源。</span><span class="sxs-lookup"><span data-stu-id="dca5d-113">For example, the network might be down, communications could be interrupted, a remote service might be unresponsive or in an unstable state, or a remote resource might be temporarily inaccessible, perhaps due to resource constraints.</span></span> <span data-ttu-id="dca5d-114">在许多情况下，故障是暂时的，可使用[重试模式][retry-pattern]处理。</span><span class="sxs-lookup"><span data-stu-id="dca5d-114">In many cases the failures will be transient and can be handled by using the [Retry pattern][retry-pattern].</span></span>

<span data-ttu-id="dca5d-115">如果应用程序检测到一个更持久的故障且无法轻松地从中恢复，则它必须能够将系统还原到一致状态，并确保整个操作的完整性。</span><span class="sxs-lookup"><span data-stu-id="dca5d-115">If the application detects a more permanent fault it can't easily recover from, it must be able to restore the system to a consistent state and ensure integrity of the entire operation.</span></span>

## <a name="solution"></a><span data-ttu-id="dca5d-116">解决方案</span><span class="sxs-lookup"><span data-stu-id="dca5d-116">Solution</span></span>

<span data-ttu-id="dca5d-117">计划程序代理监督程序模式定义了以下参与者。</span><span class="sxs-lookup"><span data-stu-id="dca5d-117">The Scheduler Agent Supervisor pattern defines the following actors.</span></span> <span data-ttu-id="dca5d-118">这些参与者对各个步骤进行协调，使其作为整个任务的一部分执行。</span><span class="sxs-lookup"><span data-stu-id="dca5d-118">These actors orchestrate the steps to be performed as part of the overall task.</span></span>

- <span data-ttu-id="dca5d-119">**计划程序**安排执行构成任务的各个步骤并协调其操作。</span><span class="sxs-lookup"><span data-stu-id="dca5d-119">The **Scheduler** arranges for the steps that make up the task to be executed and orchestrates their operation.</span></span> <span data-ttu-id="dca5d-120">这些步骤可以合并到管道或工作流中。</span><span class="sxs-lookup"><span data-stu-id="dca5d-120">These steps can be combined into a pipeline or workflow.</span></span> <span data-ttu-id="dca5d-121">计划程序负责确保按正确顺序执行此工作流中的步骤。</span><span class="sxs-lookup"><span data-stu-id="dca5d-121">The Scheduler is responsible for ensuring that the steps in this workflow are performed in the right order.</span></span> <span data-ttu-id="dca5d-122">执行每个步骤时，计划程序会记录工作流的状态，比如“步骤尚未启动”、“步骤正在运行”或“步骤已完成”。</span><span class="sxs-lookup"><span data-stu-id="dca5d-122">As each step is performed, the Scheduler records the state of the workflow, such as "step not yet started," "step running," or "step completed."</span></span> <span data-ttu-id="dca5d-123">状态信息还应包括完成该步骤所允许的时间上限，即最迟完成时间。</span><span class="sxs-lookup"><span data-stu-id="dca5d-123">The state information should also include an upper limit of the time allowed for the step to finish, called the complete-by time.</span></span> <span data-ttu-id="dca5d-124">如果某个步骤要求访问远程服务或资源，计划程序会调用相应的代理，向其传递要执行的工作的详细信息。</span><span class="sxs-lookup"><span data-stu-id="dca5d-124">If a step requires access to a remote service or resource, the Scheduler invokes the appropriate Agent, passing it the details of the work to be performed.</span></span> <span data-ttu-id="dca5d-125">计划程序通常使用异步请求/响应消息传送与代理通信。</span><span class="sxs-lookup"><span data-stu-id="dca5d-125">The Scheduler typically communicates with an Agent using asynchronous request/response messaging.</span></span> <span data-ttu-id="dca5d-126">这可以使用队列实现，不过也可改用其他分布式消息传送技术。</span><span class="sxs-lookup"><span data-stu-id="dca5d-126">This can be implemented using queues, although other distributed messaging technologies could be used instead.</span></span>

    > <span data-ttu-id="dca5d-127">计划程序执行的功能与[进程管理器模式](http://www.enterpriseintegrationpatterns.com/patterns/messaging/ProcessManager.html)下的进程管理器相似。</span><span class="sxs-lookup"><span data-stu-id="dca5d-127">The Scheduler performs a similar function to the Process Manager in the [Process Manager pattern](http://www.enterpriseintegrationpatterns.com/patterns/messaging/ProcessManager.html).</span></span> <span data-ttu-id="dca5d-128">实际的工作流通常由计划程序所控制的工作流引擎来定义和实施。</span><span class="sxs-lookup"><span data-stu-id="dca5d-128">The actual workflow is typically defined and implemented by a workflow engine that's controlled by the Scheduler.</span></span> <span data-ttu-id="dca5d-129">此方法将工作流中的业务逻辑与计划程序相分离。</span><span class="sxs-lookup"><span data-stu-id="dca5d-129">This approach decouples the business logic in the workflow from the Scheduler.</span></span>

- <span data-ttu-id="dca5d-130">**代理**包含一种逻辑，该逻辑封装对任务中某个步骤引用的远程服务的调用或远程资源的访问。</span><span class="sxs-lookup"><span data-stu-id="dca5d-130">The **Agent** contains logic that encapsulates a call to a remote service, or access to a remote resource referenced by a step in a task.</span></span> <span data-ttu-id="dca5d-131">通常，每个代理均包装对单个服务或资源的调用，并实现相应的错误处理和重试逻辑（受后面所述的超时约束限制）。</span><span class="sxs-lookup"><span data-stu-id="dca5d-131">Each Agent typically wraps calls to a single service or resource, implementing the appropriate error handling and retry logic (subject to a timeout constraint, described later).</span></span> <span data-ttu-id="dca5d-132">如果计划程序正在运行的工作流中的步骤跨不同步骤使用多个服务和资源，则每个步骤可能引用一个不同的代理（这是此模式的实现细节）。</span><span class="sxs-lookup"><span data-stu-id="dca5d-132">If the steps in the workflow being run by the Scheduler use several services and resources across different steps, each step might reference a different Agent (this is an implementation detail of the pattern).</span></span>

- <span data-ttu-id="dca5d-133">**监督程序**监视由计划程序执行的任务中步骤的状态。</span><span class="sxs-lookup"><span data-stu-id="dca5d-133">The **Supervisor** monitors the status of the steps in the task being performed by the Scheduler.</span></span> <span data-ttu-id="dca5d-134">它定期运行（频率因系统而异），并检查由计划程序维护的步骤的状态。</span><span class="sxs-lookup"><span data-stu-id="dca5d-134">It runs periodically (the frequency will be system specific), and examines the status of steps maintained by the Scheduler.</span></span> <span data-ttu-id="dca5d-135">如果它检测到任何超时或失败的步骤，则会安排相应的代理来恢复该步骤或执行相应的更正措施（这可能涉及修改步骤的状态）。</span><span class="sxs-lookup"><span data-stu-id="dca5d-135">If it detects any that have timed out or failed, it arranges for the appropriate Agent to recover the step or execute the appropriate remedial action (this might involve modifying the status of a step).</span></span> <span data-ttu-id="dca5d-136">请注意，恢复或更正操作由计划程序和代理实施。</span><span class="sxs-lookup"><span data-stu-id="dca5d-136">Note that the recovery or remedial actions are implemented by the Scheduler and Agents.</span></span> <span data-ttu-id="dca5d-137">监督程序应只请求执行这些操作。</span><span class="sxs-lookup"><span data-stu-id="dca5d-137">The Supervisor should simply request that these actions be performed.</span></span>

<span data-ttu-id="dca5d-138">计划程序、代理和监督程序都是逻辑组件，其物理实现依赖于所采用的技术。</span><span class="sxs-lookup"><span data-stu-id="dca5d-138">The Scheduler, Agent, and Supervisor are logical components and their physical implementation depends on the technology being used.</span></span> <span data-ttu-id="dca5d-139">例如，多个逻辑代理可能作为单个 Web 服务的一部分实现。</span><span class="sxs-lookup"><span data-stu-id="dca5d-139">For example, several logical agents might be implemented as part of a single web service.</span></span>

<span data-ttu-id="dca5d-140">计划程序在一个名为状态存储的持久数据存储中维护有关任务进度和各步骤的状态的信息。</span><span class="sxs-lookup"><span data-stu-id="dca5d-140">The Scheduler maintains information about the progress of the task and the state of each step in a durable data store, called the state store.</span></span> <span data-ttu-id="dca5d-141">监督程序可以使用此信息帮助确定某个步骤是否失败。</span><span class="sxs-lookup"><span data-stu-id="dca5d-141">The Supervisor can use this information to help determine whether a step has failed.</span></span> <span data-ttu-id="dca5d-142">下图阐释了计划程序、代理、监督程序和状态存储之间的关系。</span><span class="sxs-lookup"><span data-stu-id="dca5d-142">The figure illustrates the relationship between the Scheduler, the Agents, the Supervisor, and the state store.</span></span>

> <span data-ttu-id="dca5d-144">此图展示了该模式的简化版本。</span><span class="sxs-lookup"><span data-stu-id="dca5d-144">This diagram shows a simplified version of the pattern.</span></span> <span data-ttu-id="dca5d-145">在实际实施中，可能会同时运行计划程序的多个实例，每个任务子集一个实例。</span><span class="sxs-lookup"><span data-stu-id="dca5d-145">In a real implementation, there might be many instances of the Scheduler running concurrently, each a subset of tasks.</span></span> <span data-ttu-id="dca5d-146">同样，系统也可以运行每个代理的多个实例，甚至运行多个监督程序。</span><span class="sxs-lookup"><span data-stu-id="dca5d-146">Similarly, the system could run multiple instances of each Agent, or even multiple Supervisors.</span></span> <span data-ttu-id="dca5d-147">在这种情况下，各监督程序之间必须谨慎协调工作，确保它们不会争相恢复相同的失败步骤和任务。</span><span class="sxs-lookup"><span data-stu-id="dca5d-147">In this case, Supervisors must coordinate their work with each other carefully to ensure that they don’t compete to recover the same failed steps and tasks.</span></span> <span data-ttu-id="dca5d-148">[领导选拔模式](leader-election.md)为此问题提供了一个可行的解决方案。</span><span class="sxs-lookup"><span data-stu-id="dca5d-148">The [Leader Election pattern](leader-election.md) provides one possible solution to this problem.</span></span>

<span data-ttu-id="dca5d-149">当应用程序准备好运行某个任务时，它会向计划程序提交请求。</span><span class="sxs-lookup"><span data-stu-id="dca5d-149">When the application is ready to run a task, it submits a request to the Scheduler.</span></span> <span data-ttu-id="dca5d-150">计划程序在状态存储中记录有关该任务及其步骤的初始状态信息（例如，步骤尚未启动），然后开始执行工作流定义的操作。</span><span class="sxs-lookup"><span data-stu-id="dca5d-150">The Scheduler records initial state information about the task and its steps (for example, step not yet started) in the state store and then starts performing the operations defined by the workflow.</span></span> <span data-ttu-id="dca5d-151">在计划程序启动每个步骤时，它会更新状态存储中有关该步骤状态的信息（例如，步骤正在运行）。</span><span class="sxs-lookup"><span data-stu-id="dca5d-151">As the Scheduler starts each step, it updates the information about the state of that step in the state store (for example, step running).</span></span>

<span data-ttu-id="dca5d-152">如果步骤引用远程服务或资源，计划程序会将消息发送到相应的代理。</span><span class="sxs-lookup"><span data-stu-id="dca5d-152">If a step references a remote service or resource, the Scheduler sends a message to the appropriate Agent.</span></span> <span data-ttu-id="dca5d-153">除了操作的最迟完成时间外，该消息还包含代理传递到服务或访问资源所需的信息。</span><span class="sxs-lookup"><span data-stu-id="dca5d-153">The message contains the information that the Agent needs to pass to the service or access the resource, in addition to the complete-by time for the operation.</span></span> <span data-ttu-id="dca5d-154">如果代理成功完成其操作，则向计划程序返回一个响应。</span><span class="sxs-lookup"><span data-stu-id="dca5d-154">If the Agent completes its operation successfully, it returns a response to the Scheduler.</span></span> <span data-ttu-id="dca5d-155">接着，计划程序可以更新状态存储中的状态信息（例如，步骤已完成），并执行下一个步骤。</span><span class="sxs-lookup"><span data-stu-id="dca5d-155">The Scheduler can then update the state information in the state store (for example, step completed) and perform the next step.</span></span> <span data-ttu-id="dca5d-156">此过程会一直持续到整个任务完成。</span><span class="sxs-lookup"><span data-stu-id="dca5d-156">This process continues until the entire task is complete.</span></span>

<span data-ttu-id="dca5d-157">代理可以实现执行其工作所需的任何重试逻辑。</span><span class="sxs-lookup"><span data-stu-id="dca5d-157">An Agent can implement any retry logic that's necessary to perform its work.</span></span> <span data-ttu-id="dca5d-158">但是，如果在最迟完成时间到期前，代理未完成其工作，计划程序将假定该操作失败。</span><span class="sxs-lookup"><span data-stu-id="dca5d-158">However, if the Agent doesn't complete its work before the complete-by period expires, the Scheduler will assume that the operation has failed.</span></span> <span data-ttu-id="dca5d-159">在这种情况下，代理应停止其工作，不再尝试向计划程序返回任何信息（甚至包括错误消息）或尝试执行任何形式的恢复。</span><span class="sxs-lookup"><span data-stu-id="dca5d-159">In this case, the Agent should stop its work and not try to return anything to the Scheduler (not even an error message), or try any form of recovery.</span></span> <span data-ttu-id="dca5d-160">设置此限制的原因是，某个步骤超时或失败后，可能会安排代理的另一个实例来运行失败的步骤（后面介绍了此过程）。</span><span class="sxs-lookup"><span data-stu-id="dca5d-160">The reason for this restriction is that, after a step has timed out or failed, another instance of the Agent might be scheduled to run the failing step (this process is described later).</span></span>

<span data-ttu-id="dca5d-161">如果代理失败，计划程序不会收到响应。</span><span class="sxs-lookup"><span data-stu-id="dca5d-161">If the Agent fails, the Scheduler won't receive a response.</span></span> <span data-ttu-id="dca5d-162">该模式不会区分超时的步骤和真正失败的步骤。</span><span class="sxs-lookup"><span data-stu-id="dca5d-162">The pattern doesn't make a distinction between a step that has timed out and one that has genuinely failed.</span></span>

<span data-ttu-id="dca5d-163">如果某个步骤超时或失败，状态存储将包含一条记录，指示该步骤正在运行，但已超过最迟完成时间。</span><span class="sxs-lookup"><span data-stu-id="dca5d-163">If a step times out or fails, the state store will contain a record that indicates that the step is running, but the complete-by time will have passed.</span></span> <span data-ttu-id="dca5d-164">监督程序会查找类似的步骤，并尝试恢复它们。</span><span class="sxs-lookup"><span data-stu-id="dca5d-164">The Supervisor looks for steps like this and tries to recover them.</span></span> <span data-ttu-id="dca5d-165">一个可行的策略是让监督程序更新最迟完成时间值，将时间延长到足够完成该步骤，然后将消息发送到确定该步骤已超时的计划程序。接着，计划程序可以尝试重复此步骤。</span><span class="sxs-lookup"><span data-stu-id="dca5d-165">One possible strategy is for the Supervisor to update the complete-by value to extend the time available to complete the step, and then send a message to the Scheduler identifying the step that has timed out. The Scheduler can then try to repeat this step.</span></span> <span data-ttu-id="dca5d-166">但是，此设计要求任务是幂等的。</span><span class="sxs-lookup"><span data-stu-id="dca5d-166">However, this design requires the tasks to be idempotent.</span></span>

<span data-ttu-id="dca5d-167">如果同一步骤不断失败或超时，监督程序可能需要阻止重试该步骤。为此，监督程序可以在状态存储中为每个步骤维护重试计数和状态信息。</span><span class="sxs-lookup"><span data-stu-id="dca5d-167">The Supervisor might need to prevent the same step from being retried if it continually fails or times out. To do this, the Supervisor could maintain a retry count for each step, along with the state information, in the state store.</span></span> <span data-ttu-id="dca5d-168">如果此计数超过预定义的阈值，监督程序可以采取以下策略：在通知计划程序应重试该步骤前，再多等一段时间，以期故障在这段时间内得到解决。</span><span class="sxs-lookup"><span data-stu-id="dca5d-168">If this count exceeds a predefined threshold the Supervisor can adopt a strategy of waiting for an extended period before notifying the Scheduler that it should retry the step, in the expectation that the fault will be resolved during this period.</span></span> <span data-ttu-id="dca5d-169">或者，监督程序也可以向计划程序发送消息，请求通过实施[补偿事务模式](compensating-transaction.md)来撤消整个任务。</span><span class="sxs-lookup"><span data-stu-id="dca5d-169">Alternatively, the Supervisor can send a message to the Scheduler to request the entire task be undone by implementing a [Compensating Transaction pattern](compensating-transaction.md).</span></span> <span data-ttu-id="dca5d-170">此方法依赖于计划程序和代理提供必要的信息，为成功完成的每个步骤实施补偿操作。</span><span class="sxs-lookup"><span data-stu-id="dca5d-170">This approach will depend on the Scheduler and Agents providing the information necessary to implement the compensating operations for each step that completed successfully.</span></span>
