## <a name="application-design"></a><span data-ttu-id="9163d-106">应用程序设计</span><span class="sxs-lookup"><span data-stu-id="9163d-106">Application design</span></span>

<span data-ttu-id="9163d-107">**对工作负荷进行分区**。</span><span class="sxs-lookup"><span data-stu-id="9163d-107">**Partition the workload**.</span></span> <span data-ttu-id="9163d-108">将流程的各部分设计为相互独立且可以分解。</span><span class="sxs-lookup"><span data-stu-id="9163d-108">Design parts of the process to be discrete and decomposable.</span></span> <span data-ttu-id="9163d-109">最大程度地缩小各部分的大小，同时应遵循进行问题隔离所需的常用规则，以及单独责任原则。</span><span class="sxs-lookup"><span data-stu-id="9163d-109">Minimize the size of each part, while following the usual rules for separation of concerns and the single responsibility principle.</span></span> <span data-ttu-id="9163d-110">这样可以让组件的各部分采用适当的方式进行分布，从而最大程度地利用每个计算单元（例如角色或数据库服务器）。</span><span class="sxs-lookup"><span data-stu-id="9163d-110">This allows the component parts to be distributed in a way that maximizes use of each compute unit (such as a role or database server).</span></span> <span data-ttu-id="9163d-111">此外，可以更方便地缩放应用程序（只需添加特定资源的更多实例即可）。</span><span class="sxs-lookup"><span data-stu-id="9163d-111">It also makes it easier to scale the application by adding instances of specific resources.</span></span> <span data-ttu-id="9163d-112">对于复杂的域，请考虑采用[微服务体系结构](..//guide/architecture-styles/microservices.md)。</span><span class="sxs-lookup"><span data-stu-id="9163d-112">For complex domains, consider adopting a [microservices architecture](..//guide/architecture-styles/microservices.md).</span></span>

<span data-ttu-id="9163d-113">**针对缩放进行设计**。</span><span class="sxs-lookup"><span data-stu-id="9163d-113">**Design for scaling**.</span></span> <span data-ttu-id="9163d-114">缩放可以让应用程序应对不同的负载，只需增加和减小角色、队列和所使用的其他服务的实例数目即可。</span><span class="sxs-lookup"><span data-stu-id="9163d-114">Scaling allows applications to react to variable load by increasing and decreasing the number of instances of roles, queues, and other services they use.</span></span> <span data-ttu-id="9163d-115">但是，应用程序在设计时必须充分考虑这一点。</span><span class="sxs-lookup"><span data-stu-id="9163d-115">However, the application must be designed with this in mind.</span></span> <span data-ttu-id="9163d-116">例如，应用程序及其使用的服务必须是无状态的，这样才能将请求路由到任何实例。</span><span class="sxs-lookup"><span data-stu-id="9163d-116">For example, the application and the services it uses must be stateless, to allow requests to be routed to any instance.</span></span> <span data-ttu-id="9163d-117">此外，使得添加或删除特定的实例不会对当前用户造成不利影响。</span><span class="sxs-lookup"><span data-stu-id="9163d-117">This also prevents the addition or removal of specific instances from adversely impacting current users.</span></span> <span data-ttu-id="9163d-118">还应该在添加和删除实例时实现实例的配置或对其进行自动检测，使得应用程序中的代码能够执行必需的路由操作。</span><span class="sxs-lookup"><span data-stu-id="9163d-118">You should also implement configuration or auto-detection of instances as they are added and removed, so that code in the application can perform the necessary routing.</span></span> <span data-ttu-id="9163d-119">例如，Web 应用程序可能会通过轮循方法使用一组队列来路由请求，将请求路由到通过辅助角色运行的后台服务。</span><span class="sxs-lookup"><span data-stu-id="9163d-119">For example, a web application might use a set of queues in a class="sxs-lookup"><span data-stu-id="9163d-120">The web application must be able to detect changes in the number of queues, to successfully route requests and balance the load on the <span data-ttu-id="9163d-121">**按单元进行缩放**。</span><span class="sxs-lookup"><span data-stu-id="9163d-121">**Scale as a unit**.</span></span> <span data-ttu-id="9163d-122">为了应对业务增长情况，需要按计划添加更多资源。</span><span class="sxs-lookup"><span data-stu-id="9163d-122">Plan for additional resources to accommodate growth.</span></span> <span data-ttu-id="9163d-123">对于每个资源，应该知道缩放上限，并使用分片或分解方式来应对这些限制。</span><span class="sxs-lookup"><span data-stu-id="9163d-123">For each resource, know the upper scaling limits, and use sharding or decomposition to go beyond these limits.</span></span> <span data-ttu-id="9163d-124">按照定义好的资源集来确定系统的缩放单元。</span><span class="sxs-lookup"><span data-stu-id="9163d-124">Determine the scale units for the system in terms of well-defined sets of resources.</span></span> <span data-ttu-id="9163d-125">这会使横向扩展操作更加容易，不容易对应用程序造成不利影响（整个系统的某些部分缺乏资源时，会对应用程序施加限制）。</span><span class="sxs-lookup"><span data-stu-id="9163d-125">This makes applying scale-out operations easier, and less prone to negative impact on the application through limitations imposed by lack of resources in some part of the overall system.</span></span> <span data-ttu-id="9163d-126">例如，添加 x 个 Web 和辅助角色可能需要使用 y 个额外的队列和 z 个存储帐户来处理这些角色所造成的额外工作负荷。</span><span class="sxs-lookup"><span data-stu-id="9163d-126">For example, adding x number of web and worker roles might require y number of additional queues and z number of storage accounts to handle the additional workload generated by the roles.</span></span> <span data-ttu-id="9163d-127">因此缩放单位可能包含 x 个 Web 和辅助角色、*y* 个队列和 *z* 个存储帐户。</span><span class="sxs-lookup"><span data-stu-id="9163d-127">So a scale unit could consist of x web and worker roles, *y* queues, and *z* storage accounts.</span></span> <span data-ttu-id="9163d-128">设计应用程序时，应确保其能够轻松地通过添加一个或多个缩放单元来进行缩放。</span><span class="sxs-lookup"><span data-stu-id="9163d-128">Design the application so that it's easily scaled by adding one or more scale units.</span></span>

<span data-ttu-id="9163d-129">**避免客户端关联**。</span><span class="sxs-lookup"><span data-stu-id="9163d-129">**Avoid client affinity**.</span></span> <span data-ttu-id="9163d-130">在可能的情况下，应确保应用程序不需要关联。</span><span class="sxs-lookup"><span data-stu-id="9163d-130">Where possible, ensure that the application does not require affinity.</span></span> <span data-ttu-id="9163d-131">因此，请求能够路由到任何实例，与实例的数目无关。</span><span class="sxs-lookup"><span data-stu-id="9163d-131">Requests can thus be routed to any instance, and the number of instances is irrelevant.</span></span> <span data-ttu-id="9163d-132">这样还可以避免为每个用户存储、检索和维护状态信息的开销。</span><span class="sxs-lookup"><span data-stu-id="9163d-132">This also avoids the overhead of storing, retrieving, and maintaining state information for each user.</span></span>

<span data-ttu-id="9163d-133">**充分利用平台自动缩放功能**。</span><span class="sxs-lookup"><span data-stu-id="9163d-133">**Take advantage of platform autoscaling features**.</span></span> <span data-ttu-id="9163d-134">在托管平台支持自动缩放功能（例如 Azure 自动缩放）的情况下，应尽量使用该功能而不使用自定义机制或第三方机制，除非内在机制无法满足要求。</span><span class="sxs-lookup"><span data-stu-id="9163d-134">Where the hosting platform supports an autoscaling capability, such as Azure Autoscale, prefer it to custom or third-party mechanisms unless the built-in mechanism can't fulfill your requirements.</span></span> <span data-ttu-id="9163d-135">尽可能使用计划的缩放规则，确保资源在开始使用时没有延迟，但应根据需要向规则中添加反应性的自动缩放，以应对需求方面的意外变化。</span><span class="sxs-lookup"><span data-stu-id="9163d-135">Use scheduled scaling rules where possible to ensure resources are available without a start-up delay, but add reactive autoscaling to the rules where appropriate to cope with unexpected changes in demand.</span></span> <span data-ttu-id="9163d-136">可以在服务管理 API 中使用自动缩放操作来调整自动缩放，并可向规则中添加自定义计数器。</span><span class="sxs-lookup"><span data-stu-id="9163d-136">You can use the autoscaling operations in the Service Management API to adjust autoscaling, and to add custom counters to rules.</span></span> <span data-ttu-id="9163d-137">有关详细信息，请参阅 [Auto-scaling guidance](../best-practices/auto-scaling.md)（自动缩放指南）。</span><span class="sxs-lookup"><span data-stu-id="9163d-137">For more information, see [Auto-scaling guidance](../best-practices/auto-scaling.md).</span></span>

<span data-ttu-id="9163d-138">**卸载密集型 CPU/IO 任务（充当后台任务）**。</span><span class="sxs-lookup"><span data-stu-id="9163d-138">**Offload intensive CPU/IO tasks as background tasks**.</span></span> <span data-ttu-id="9163d-139">如果某项服务请求预计需要较长时间才能运行完毕或需要使用大量的资源，则应将该请求的处理操作卸载到单独的任务中。</span><span class="sxs-lookup"><span data-stu-id="9163d-139">If a request to a service is expected to take a long time to run or absorb considerable resources, offload the processing for this request to a separate task.</span></span> <span data-ttu-id="9163d-140">使用辅助角色或后台作业（具体取决于托管平台）来执行这些任务。</span><span class="sxs-lookup"><span data-stu-id="9163d-140">Use worker roles or background jobs (depending on the hosting platform) to execute these tasks.</span></span> <span data-ttu-id="9163d-141">此策略使服务能够继续接收更多的请求并保持响应状态。</span><span class="sxs-lookup"><span data-stu-id="9163d-141">This strategy enables the service to continue receiving further requests and remain responsive.</span></span>  <span data-ttu-id="9163d-142">有关详细信息，请参阅[后台作业指南](../best-practices/background-jobs.md)。</span><span class="sxs-lookup"><span data-stu-id="9163d-142">For more information, see [Background jobs guidance](../best-practices/background-jobs.md).</span></span>

<span data-ttu-id="9163d-143">**分发后台任务的工作负荷**。</span><span class="sxs-lookup"><span data-stu-id="9163d-143">**Distribute the workload for background tasks**.</span></span> <span data-ttu-id="9163d-144">存在许多后台任务时，或者这些任务需要相当长的时间或相当多的资源时，可将这些工作分散到多个计算单元（例如辅助角色或后台作业）中。</span><span class="sxs-lookup"><span data-stu-id="9163d-144">Where there are many background tasks, or the tasks require considerable time or resources, spread the work across multiple compute units (such as worker roles or background jobs).</span></span> <span data-ttu-id="9163d-145">如需一种可能的解决方案，请参阅 [Competing Consumers Pattern](https://msdn.microsoft.com/library/dn568101.aspx)（使用者竞争模式）。</span><span class="sxs-lookup"><span data-stu-id="9163d-145">For one possible solution, see the [Competing Consumers Pattern](https://msdn.microsoft.com/library/dn568101.aspx).</span></span>

<span data-ttu-id="9163d-146">**考虑转移到*非共享*体系结构**。</span><span class="sxs-lookup"><span data-stu-id="9163d-146">**Consider moving towards a *shared-nothing* architecture**.</span></span> <span data-ttu-id="9163d-147">非共享体系结构使用独立的和自足的节点，这些节点不存在争用情况（例如共享服务或存储）。</span><span class="sxs-lookup"><span data-stu-id="9163d-147">A shared-nothing architecture uses independent, self-sufficient nodes that have no single point of contention (such as shared services or storage).</span></span> <span data-ttu-id="9163d-148">从理论上讲，此类系统几乎可以无限地进行扩展。</span><span class="sxs-lookup"><span data-stu-id="9163d-148">In theory, such a system can scale almost indefinitely.</span></span> <span data-ttu-id="9163d-149">虽然完全的非共享方法对于大多数应用程序来说并不实际，但这提供了一种思路，即如何进行设计才能获得更好的缩放性。</span><span class="sxs-lookup"><span data-stu-id="9163d-149">While a fully shared-nothing approach is generally not practical for most applications, it may provide opportunities to design for better scalability.</span></span> <span data-ttu-id="9163d-150">例如，避免使用服务器端会话状态、客户端关联、数据分区，这样可以构造一个近乎没有共享的体系结构。</span><span class="sxs-lookup"><span data-stu-id="9163d-150">For example, avoiding the use of server-side session state, client affinity, and data partitioning are good examples of moving towards a shared-nothing architecture.</span></span>

## <a name="data-management"></a><span data-ttu-id="9163d-151">数据管理</span><span class="sxs-lookup"><span data-stu-id="9163d-151">Data management</span></span>

<span data-ttu-id="9163d-152">**使用数据分区**。</span><span class="sxs-lookup"><span data-stu-id="9163d-152">**Use data partitioning**.</span></span> <span data-ttu-id="9163d-153">将数据划分到多个数据库和数据库服务器中，或者将应用程序设计为使用数据存储服务，而该服务能够以透明方式提供此分区功能（此类例子包括 Azure SQL 数据库弹性数据库和 Azure 表存储）。</span><span class="sxs-lookup"><span data-stu-id="9163d-153">Divide the data across multiple databases and database servers, or design the application to use data storage services that can provide this partitioning transparently (examples include Azure SQL Database Elastic Database, and Azure Table storage).</span></span> <span data-ttu-id="9163d-154">这种方法可以使性能最大化，并使缩放更轻松。</span><span class="sxs-lookup"><span data-stu-id="9163d-154">This approach can help to maximize performance and allow easier scaling.</span></span> <span data-ttu-id="9163d-155">存在各种不同的分区技术，例如水平分区、垂直分区和功能分区。</span><span class="sxs-lookup"><span data-stu-id="9163d-155">There are different partitioning techniques, such as horizontal, vertical, and functional.</span></span> <span data-ttu-id="9163d-156">可以将这些技术组合起来使用，以便实现最佳效益，例如提高查询性能、简化缩放、提高管理的灵活性、提高可用性，以及使存储类型与所要存储的数据更匹配。</span><span class="sxs-lookup"><span data-stu-id="9163d-156">You can use a combination of these to achieve maximum benefit from increased query performance, simpler scalability, more flexible management, better availability, and to match the type of store to the data it will hold.</span></span> <span data-ttu-id="9163d-157">另外，还需要根据不同类型的数据来使用不同类型的数据存储，而在选择存储类型时，还要看这些类型可以针对特定类型的数据优化到何种程度。</span><span class="sxs-lookup"><span data-stu-id="9163d-157">Also, consider using different types of data store for different types of data, choosing the types based on how well they are optimized for the specific type of data.</span></span> <span data-ttu-id="9163d-158">例如，可以使用表存储、文档数据库或列系列的数据存储来代替关系数据库，或者在使用时像重视关系数据库一样重视这些数据库。</span><span class="sxs-lookup"><span data-stu-id="9163d-158">This may include using table storage, a document database, or a column-family data store, instead of, or as well as, a relational database.</span></span> <span data-ttu-id="9163d-159">有关详细信息，请参阅[数据分区指南](../best-practices/data-partitioning.md)。</span><span class="sxs-lookup"><span data-stu-id="9163d-159">For more information, see [Data partitioning guidance](../best-practices/data-partitioning.md).</span></span>

<span data-ttu-id="9163d-160">**在设计时需要考虑最终一致性**。</span><span class="sxs-lookup"><span data-stu-id="9163d-160">**Design for eventual consistency**.</span></span> <span data-ttu-id="9163d-161">最终一致性可以改进缩放性能，因为在分区跨多个存储时它可以减少进行相关数据的同步所需的时间，或者根本不需要此方面的时间。</span><span class="sxs-lookup"><span data-stu-id="9163d-161">Eventual consistency improves scalability by reducing or removing the time needed to synchronize related data partitioned across multiple stores.</span></span> <span data-ttu-id="9163d-162">代价是，数据在读取时并不是始终一致的，有时写入操作也会导致冲突。</span><span class="sxs-lookup"><span data-stu-id="9163d-162">The cost is that data is not always consistent when it is read, and some write operations may cause conflicts.</span></span> <span data-ttu-id="9163d-163">在需要频繁读取相同的数据而不需要频繁写入这些数据的情况下，最终一致性是理想的标准。</span><span class="sxs-lookup"><span data-stu-id="9163d-163">Eventual consistency is ideal for situations where the same data is read frequently but written infrequently.</span></span> <span data-ttu-id="9163d-164">有关详细信息，请参阅 [Data Consistency Primer](https://msdn.microsoft.com/library/dn589800.aspx)（数据一致性入门）。</span><span class="sxs-lookup"><span data-stu-id="9163d-164">For more information, see the [Data Consistency Primer](https://msdn.microsoft.com/library/dn589800.aspx).</span></span>

<span data-ttu-id="9163d-165">**减少组件和服务之间的聊天式交互**。</span><span class="sxs-lookup"><span data-stu-id="9163d-165">**Reduce chatty interactions between components and services**.</span></span> <span data-ttu-id="9163d-166">避免设计交互功能，防止应用程序向服务进行多次调用（每次调用返回少量数据）而不是单次调用（一次调用返回所有数据）。</span><span class="sxs-lookup"><span data-stu-id="9163d-166">Avoid designing interactions in which an application is required to make multiple calls to a service (each of which returns a small amount of data), rather than a single call that can return all of the data.</span></span> <span data-ttu-id="9163d-167">调用具有明显延迟的服务或组件时，尽可能将多个相关的操作组合到单个请求中。</span><span class="sxs-lookup"><span data-stu-id="9163d-167">Where possible, combine several related operations into a single request when the call is to a service or component that has noticeable latency.</span></span> <span data-ttu-id="9163d-168">这可以更轻松地监视性能并优化复杂操作。</span><span class="sxs-lookup"><span data-stu-id="9163d-168">This makes it easier to monitor performance and optimize complex operations.</span></span> <span data-ttu-id="9163d-169">例如，使用数据库中存储的过程来封装复杂的逻辑，减少往返次数和资源锁定次数。</span><span class="sxs-lookup"><span data-stu-id="9163d-169">For example, use stored procedures in databases to encapsulate complex logic, and reduce the number of round trips and resource locking.</span></span>

<span data-ttu-id="9163d-170">**使用队列来平衡高速数据写入的负载**。</span><span class="sxs-lookup"><span data-stu-id="9163d-170">**Use queues to level the load for high velocity data writes**.</span></span> <span data-ttu-id="9163d-171">对某项服务的需求激增可能会导致该服务不堪重负，引发越来越严重的故障。</span><span class="sxs-lookup"><span data-stu-id="9163d-171">Surges in demand for a service can overwhelm that service and cause escalating failures.</span></span> <span data-ttu-id="9163d-172">为了防止这种情况，可考虑实施[基于队列的负载均衡模式](https://msdn.microsoft.com/library/dn589783.aspx)。</span><span class="sxs-lookup"><span data-stu-id="9163d-172">To prevent this, consider implementing the [Queue-Based Load Leveling Pattern](https://msdn.microsoft.com/library/dn589783.aspx).</span></span> <span data-ttu-id="9163d-173">使用队列在任务与所调用的服务之间充当缓冲。</span><span class="sxs-lookup"><span data-stu-id="9163d-173">Use a queue that acts as a buffer between a task and a service that it invokes.</span></span> <span data-ttu-id="9163d-174">这可以平缓暂时性的超载，否则可能导致服务故障或任务超时。</span><span class="sxs-lookup"><span data-stu-id="9163d-174">This can smooth intermittent heavy loads that may otherwise cause the service to fail or the task to time out.</span></span>

## <a name="implementation"></a><span data-ttu-id="9163d-222">实现</span><span class="sxs-lookup"><span data-stu-id="9163d-222">Implementation</span></span>

<span data-ttu-id="9163d-223">**查看性能对立模式**。</span><span class="sxs-lookup"><span data-stu-id="9163d-223">**Review the performance antipatterns**.</span></span> <span data-ttu-id="9163d-224">请参阅[云应用程序的性能对立模式](../antipatterns/index.md)，了解在应用程序承受压力时，可能导致可伸缩性问题的常见做法。</span><span class="sxs-lookup"><span data-stu-id="9163d-224">See [Performance antipatterns for cloud applications](../antipatterns/index.md) for common practices that are likely to cause scalability problems when an application is under pressure.</span></span>

<span data-ttu-id="9163d-225">**使用异步调用**。</span><span class="sxs-lookup"><span data-stu-id="9163d-225">**Use asynchronous calls**.</span></span> <span data-ttu-id="9163d-226">在访问可能会受 I/O 或网络带宽限制（或者存在明显延迟）的资源或服务时，尽可能使用异步代码，以免锁定调用线程。</span><span class="sxs-lookup"><span data-stu-id="9163d-226">Use asynchronous code wherever possible when accessing resources or services that may be limited by I/O or network bandwidth, or that have a noticeable latency, in order to avoid locking the calling thread.</span></span>

<span data-ttu-id="9163d-232">**将高延迟、低带宽网络上压缩性强的数据进行压缩**。</span><span class="sxs-lookup"><span data-stu-id="9163d-232">**Compress highly compressible data over high latency, low bandwidth networks**.</span></span> <span data-ttu-id="9163d-233">在使用 Web 应用程序时，大多数情况下通过应用程序生成并通过网络传递的最大数据量来自对客户端请求的 HTTP 响应。</span><span class="sxs-lookup"><span data-stu-id="9163d-233">In the majority of cases in a web application, the largest volume of data generated by the application and passed over the network is HTTP responses to client requests.</span></span> <span data-ttu-id="9163d-234">HTTP 压缩可以大大减少此方面的数据量，尤其是对静态内容而言。</span><span class="sxs-lookup"><span data-stu-id="9163d-234">HTTP compression can reduce this considerably, especially for static content.</span></span> <span data-ttu-id="9163d-235">这可以降低成本并减少网络的负载，虽然压缩动态内容会造成服务器上的负载部分过高。</span><span class="sxs-lookup"><span data-stu-id="9163d-235">This can reduce cost as well as reducing the load on the network, though compressing dynamic content does apply a fractionally higher load on the server.</span></span> <span data-ttu-id="9163d-236">在其他更通用化的环境中，数据压缩可以降低传输的数据量，并且会最大程度地缩短传输时间和降低成本，但压缩及解压缩过程会造成开销。</span><span class="sxs-lookup"><span data-stu-id="9163d-236">In other, more generalized environments, data compression can reduce the volume of data transmitted and minimize transfer time and costs, but the compression and decompression processes incur overhead.</span></span> <span data-ttu-id="9163d-237">因此，只有在性能会明显提高的情况下，才应使用压缩。</span><span class="sxs-lookup"><span data-stu-id="9163d-237">As such, compression should only be used when there is a demonstrable gain in performance.</span></span> <span data-ttu-id="9163d-238">其他序列化方法（例如 JSON 或二进制编码）可以降低负载大小，同时对性能的影响较小，而 XML 则可能提高负载。</span><span class="sxs-lookup"><span data-stu-id="9163d-238">Other serialization methods, such as JSON or binary encodings, may reduce the payload size while having less impact on performance, whereas XML is likely to increase it.</span></span>

<span data-ttu-id="9163d-239">**将使用连接和资源的时间尽可能降至最低**。</span><span class="sxs-lookup"><span data-stu-id="9163d-239">**Minimize the time that connections and resources are in use**.</span></span> <span data-ttu-id="9163d-240">将连接和资源的维持时间降至所需的最低。</span><span class="sxs-lookup"><span data-stu-id="9163d-240">Maintain connections and resources only for as long as you need to use them.</span></span> <span data-ttu-id="9163d-241">例如，尽可能晚地打开连接，并尽可能快地将其返回到连接池。</span><span class="sxs-lookup"><span data-stu-id="9163d-241">For example, open connections as late as possible, and allow them to be returned to the connection pool as soon as possible.</span></span> <span data-ttu-id="9163d-242">尽可能晚地获取资源，并尽可能快地释放它们。</span><span class="sxs-lookup"><span data-stu-id="9163d-242">Acquire resources as late as possible, and dispose of them as soon as possible.</span></span>

<span data-ttu-id="9163d-243">**最大程度减少所需的连接数**。</span><span class="sxs-lookup"><span data-stu-id="9163d-243">**Minimize the number of connections required**.</span></span> <span data-ttu-id="9163d-244">服务连接会吸收资源。</span><span class="sxs-lookup"><span data-stu-id="9163d-244">Service connections absorb resources.</span></span> <span data-ttu-id="9163d-245">限制所需的连接数，并确保尽可能重用现有连接。</span><span class="sxs-lookup"><span data-stu-id="9163d-245">Limit the number that are required and ensure that existing connections are reused whenever possible.</span></span> <span data-ttu-id="9163d-246">例如，执行身份验证后可根据需要使用模拟，以便以特定身份来运行代码。</span><span class="sxs-lookup"><span data-stu-id="9163d-246">For example, after performing authentication, use impersonation where appropriate to run code as a specific identity.</span></span> <span data-ttu-id="9163d-247">这种方式可以重用连接，从而充分利用连接池。</span><span class="sxs-lookup"><span data-stu-id="9163d-247">This can help to make best use of the connection pool by reusing connections.</span></span>
