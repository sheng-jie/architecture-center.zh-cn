<span data-ttu-id="1782d-116">**评审并改进过程。**</span><span class="sxs-lookup"><span data-stu-id="1782d-116">**Review and improve processes.**</span></span> <span data-ttu-id="1782d-117">过程和流程（不管是自动的还是手动的）永远不是最终性的。</span><span class="sxs-lookup"><span data-stu-id="1782d-117">Your processes and procedures, both automated and manual, are never final.</span></span> <span data-ttu-id="1782d-118">建立当前工作流、流程和文档的定期评审制度，并制定持续改进的目标。</span><span class="sxs-lookup"><span data-stu-id="1782d-118">Set up regular reviews of current workflows, procedures, and documentation, with a goal of continual improvement.</span></span>

<span data-ttu-id="1782d-119">**做出前瞻性的规划。**</span><span class="sxs-lookup"><span data-stu-id="1782d-119">**Do proactive planning.**</span></span> <span data-ttu-id="1782d-120">针对故障做出前瞻性的规划。</span><span class="sxs-lookup"><span data-stu-id="1782d-120">Proactively plan for failure.</span></span> <span data-ttu-id="1782d-121">制定过程以快速识别出现的问题，将问题转达给适当的团队成员予以解决，并确认解决方法。</span><span class="sxs-lookup"><span data-stu-id="1782d-121">Have processes in place to quickly identify issues when they occur, escalate to the correct team members to fix, and confirm resolution.</span></span>

<span data-ttu-id="1782d-122">**从故障中学习经验。**</span><span class="sxs-lookup"><span data-stu-id="1782d-122">**Learn from failures.**</span></span> <span data-ttu-id="1782d-123">故障不可避免，但必须在故障中学到经验教训，以免故障重复发生。</span><span class="sxs-lookup"><span data-stu-id="1782d-123">Failures are inevitable, but it's important to learn from failures to avoid repeating them.</span></span> <span data-ttu-id="1782d-124">如果发生操作性的故障，应该围绕问题进行会审，阐述原因和解决方法，并分享学到的经验教训。</span><span class="sxs-lookup"><span data-stu-id="1782d-124">If an operational failure occurs, triage the issue, document the cause and solution, and share any lessons that were learned.</span></span> <span data-ttu-id="1782d-125">尽量更新生成过程，以便将来发生此类故障时可以自动检测得到。</span><span class="sxs-lookup"><span data-stu-id="1782d-125">Whenever possible, update your build processes to automatically detect that kind of failure in the future.</span></span>

<span data-ttu-id="1782d-126">**优化速度并收集数据。**</span><span class="sxs-lookup"><span data-stu-id="1782d-126">**Optimize for speed and collect data.**</span></span> <span data-ttu-id="1782d-127">规划的每种改进方案都是基于假设。</span><span class="sxs-lookup"><span data-stu-id="1782d-127">Every planned improvement is a hypothesis.</span></span> <span data-ttu-id="1782d-128">请尽量以最慢的步调行事。</span><span class="sxs-lookup"><span data-stu-id="1782d-128">Work in the smallest increments possible.</span></span> <span data-ttu-id="1782d-129">将新的思路视为试验。</span><span class="sxs-lookup"><span data-stu-id="1782d-129">Treat new ideas as experiments.</span></span> <span data-ttu-id="1782d-130">检测试验，以便可以收集生产数据来评估这些试验的有效性。</span><span class="sxs-lookup"><span data-stu-id="1782d-130">Instrument the experiments so that you can collect production data to assess their effectiveness.</span></span> <span data-ttu-id="1782d-131">如果假设有误，应准备好尽快从失败中恢复。</span><span class="sxs-lookup"><span data-stu-id="1782d-131">Be prepared to fail fast if the hypothesis is wrong.</span></span>

<span data-ttu-id="1782d-132">**花费足够的时间学习。**</span><span class="sxs-lookup"><span data-stu-id="1782d-132">**Allow time for learning.**</span></span> <span data-ttu-id="1782d-133">失败和成功都是很好的学习机会。</span><span class="sxs-lookup"><span data-stu-id="1782d-133">Both failures and successes provide good opportunities for learning.</span></span> <span data-ttu-id="1782d-134">在转移到新项目之前，请花费足够的时间来收集重要的课程，并确保这些课程可由团队吸收。</span><span class="sxs-lookup"><span data-stu-id="1782d-134">Before moving on to new projects, allow enough time to gather the important lessons, and make sure those lessons are absorbed by your team.</span></span> <span data-ttu-id="1782d-135">此外，为团队提供一些时间来培养技能、展开试验，以及学习新的工具和技术。</span><span class="sxs-lookup"><span data-stu-id="1782d-135">Also give the team the time to build skills, experiment, and learn about new tools and techniques.</span></span>

<span data-ttu-id="1782d-142">**分享知识。**</span><span class="sxs-lookup"><span data-stu-id="1782d-142">**Share knowledge.**</span></span> <span data-ttu-id="1782d-143">文档只对知道它存在并可以找到它的人员才有作用。</span><span class="sxs-lookup"><span data-stu-id="1782d-143">Documentation is only useful if people know that it exists and can find it.</span></span> <span data-ttu-id="1782d-144">请确保文档组织有序且能轻松找到。</span><span class="sxs-lookup"><span data-stu-id="1782d-144">Ensure the documentation is organized and easily discoverable.</span></span> <span data-ttu-id="1782d-145">创新方式：使用牛皮纸袋（非正式演示）、视频或新闻稿分享知识。</span><span class="sxs-lookup"><span data-stu-id="1782d-145">Be creative: Use brown bags (informal presentations), videos, or newsletters to share knowledge.</span></span>

## <a name="development"></a><span data-ttu-id="1782d-146">开发</span><span class="sxs-lookup"><span data-stu-id="1782d-146">Development</span></span>

<span data-ttu-id="1782d-147">**为开发人员提供类似于生产的环境。**</span><span class="sxs-lookup"><span data-stu-id="1782d-147">**Provide developers with production-like environments.**</span></span> <span data-ttu-id="1782d-148">如果开发和测试环境与生产环境不相符，则很难测试和诊断问题。</span><span class="sxs-lookup"><span data-stu-id="1782d-148">If development and test environments don't match the production environment, it is hard to test and diagnose problems.</span></span> <span data-ttu-id="1782d-149">因此，应该让开发和测试环境尽量接近生产环境。</span><span class="sxs-lookup"><span data-stu-id="1782d-149">Therefore, keep development and test environments as close to the production environment as possible.</span></span> <span data-ttu-id="1782d-150">确保测试数据与生产环境中使用的数据一致，即使测试数据是示例数据而不是实际生产数据（出于隐私或符合性原因）。</span><span class="sxs-lookup"><span data-stu-id="1782d-150">Make sure that test data is consistent with the data used in production, even if it's sample data and not real production data (for privacy or compliance reasons).</span></span> <span data-ttu-id="1782d-151">计划生成并匿名化示例测试数据。</span><span class="sxs-lookup"><span data-stu-id="1782d-151">Plan to generate and anonymize sample test data.</span></span>

<span data-ttu-id="1782d-152">**确保所有已获授权的团队成员可以预配基础结构和部署应用程序。**</span><span class="sxs-lookup"><span data-stu-id="1782d-152">**Ensure that all authorized team members can provision infrastructure and deploy the application.**</span></span> <span data-ttu-id="1782d-153">设置类似于生产环境中所用的资源和部署应用程序不应涉及复杂的手动任务，或者要求对系统有深层的技术知识。</span><span class="sxs-lookup"><span data-stu-id="1782d-153">Setting up production-like resources and deploying the application should not involve complicated manual tasks or detailed technical knowledge of the system.</span></span> <span data-ttu-id="1782d-154">具有适当权限的任何人应该能够创建或部署类似于生产环境中所用的资源，而无需求助于运营团队。</span><span class="sxs-lookup"><span data-stu-id="1782d-154">Anyone with the right permissions should be able to create or deploy production-like resources without going to the operations team.</span></span> 

> <span data-ttu-id="1782d-155">这项建议并不意味着任何人都可以将实时更新推送到生产部署。</span><span class="sxs-lookup"><span data-stu-id="1782d-155">This recommendation doesn't imply that anyone can push live updates to the production deployment.</span></span> <span data-ttu-id="1782d-156">其目的是减少开发和 QA 团队在创建类似于生产的环境时所产生的矛盾。</span><span class="sxs-lookup"><span data-stu-id="1782d-156">It's about reducing friction for the development and QA teams to create production-like environments.</span></span>

<span data-ttu-id="1782d-157">**检测应用程序以获取见解。**</span><span class="sxs-lookup"><span data-stu-id="1782d-157">**Instrument the application for insight.**</span></span> <span data-ttu-id="1782d-158">若要了解应用程序的运行状况，需要知道它的执行情况，以及是否出现了任何错误或问题。</span><span class="sxs-lookup"><span data-stu-id="1782d-158">To understand the health of your application, you need to know how it's performing and whether it's experiencing any errors or problems.</span></span> <span data-ttu-id="1782d-159">始终将检测包含为一项设计要求，并且一开始就在应用程序中内置检测功能。</span><span class="sxs-lookup"><span data-stu-id="1782d-159">Always include instrumentation as a design requirement, and build the instrumentation into the application from the start.</span></span> <span data-ttu-id="1782d-160">检测必须包括用于分析根本原因的事件日志记录，同时还应包括用于监视应用程序总体运行状况和使用情况的遥测功能与指标。</span><span class="sxs-lookup"><span data-stu-id="1782d-160">Instrumentation must include event logging for root cause analysis, but also telemetry and metrics to monitor the overall health and usage of the application.</span></span>

<span data-ttu-id="1782d-161">**跟踪技术债务。**</span><span class="sxs-lookup"><span data-stu-id="1782d-161">**Track your technical debt.**</span></span> <span data-ttu-id="1782d-162">在许多项目中，发布计划的优先级在一定程度上高于代码质量。</span><span class="sxs-lookup"><span data-stu-id="1782d-162">In many projects, release schedules can get prioritized over code quality to one degree or another.</span></span> <span data-ttu-id="1782d-163">如果存在这种情况，请始终保持跟踪。</span><span class="sxs-lookup"><span data-stu-id="1782d-163">Always keep track when this occurs.</span></span> <span data-ttu-id="1782d-164">阐述所有快捷做法或其他非最佳实施方法，并计划在将来的某个时间重新审视这些问题。</span><span class="sxs-lookup"><span data-stu-id="1782d-164">Document any shortcuts or other nonoptimal implementations, and schedule time in the future to revisit these issues.</span></span>

<span data-ttu-id="1782d-165">**考虑将更新直接推送到生产环境。**</span><span class="sxs-lookup"><span data-stu-id="1782d-165">**Consider pushing updates directly to production.**</span></span> <span data-ttu-id="1782d-166">为了缩短总体发布周期时间，请考虑将适当测试的代码提交内容直接推送到生产环境。</span><span class="sxs-lookup"><span data-stu-id="1782d-166">To reduce the overall release cycle time, consider pushing properly tested code commits directly to production.</span></span> <span data-ttu-id="1782d-167">使用[功能切换][feature-toggles]来控制要启用哪些功能。</span><span class="sxs-lookup"><span data-stu-id="1782d-167">Use [feature toggles][feature-toggles] to control which features are enabled.</span></span> <span data-ttu-id="1782d-168">这样，就可以使用切换机制启用或禁用功能，快速从开发转移到发布阶段。</span><span class="sxs-lookup"><span data-stu-id="1782d-168">This allows you to move from development to release quickly, using the toggles to enable or disable features.</span></span> <span data-ttu-id="1782d-169">执行 [Canary 发布][canary-release]（将特定的功能部署到生产环境的一部分区域）等测试时，切换机制也很有用。</span><span class="sxs-lookup"><span data-stu-id="1782d-169">Toggles are also useful when performing tests such as [canary releases][canary-release], where a particular feature is deployed to a subset of the production environment.</span></span>
