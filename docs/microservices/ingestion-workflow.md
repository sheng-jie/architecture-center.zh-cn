<span data-ttu-id="ab52a-115">这是整个应用程序的核心，因此，端到端的过程必须高效且可靠。</span><span class="sxs-lookup"><span data-stu-id="ab52a-115">This is the core of the entire application, so the end-to-end process must be performant as well as reliable.</span></span> <span data-ttu-id="ab52a-116">必须解决一些特殊的难题：</span><span class="sxs-lookup"><span data-stu-id="ab52a-116">Some particular challenges must be addressed:</span></span>

## <a name="ingestion"></a><span data-ttu-id="ab52a-131">引入</span><span class="sxs-lookup"><span data-stu-id="ab52a-131">Ingestion</span></span>

<span data-ttu-id="ab52a-132">根据业务要求，开发团队确定了与引入相关的以下非功能性要求：</span><span class="sxs-lookup"><span data-stu-id="ab52a-132">Based on business requirements, the development team identified the following non-functional requirements for ingestion:</span></span>

- <span data-ttu-id="ab52a-133">每秒 10000 个请求的持续性吞吐量。</span><span class="sxs-lookup"><span data-stu-id="ab52a-133">Sustained throughput of 10K requests/sec.</span></span>
- <span data-ttu-id="ab52a-134">在高峰期每秒能够处理多达 50000 个请求，且不会丢弃客户端请求或导致超时。</span><span class="sxs-lookup"><span data-stu-id="ab52a-134">Able to handle spikes of up to 50K/sec without dropping client requests or timing out.</span></span>
- <span data-ttu-id="ab52a-135">在 99% 的时间内延迟小于 500 毫秒。</span><span class="sxs-lookup"><span data-stu-id="ab52a-135">Less than 500ms latency in the 99th percentile.</span></span>

<span data-ttu-id="ab52a-136">处理偶发性流量高峰的要求为设计带来了挑战。</span><span class="sxs-lookup"><span data-stu-id="ab52a-136">The requirement to handle occasional spikes in traffic presents a design challenge.</span></span> <span data-ttu-id="ab52a-137">从理论上讲，系统可以横向扩展，以处理最大预期流量。</span><span class="sxs-lookup"><span data-stu-id="ab52a-137">In theory, the system could be scaled out to handle the maximum expected traffic.</span></span> <span data-ttu-id="ab52a-138">但是，预配这么多的资源是极其低效的做法。</span><span class="sxs-lookup"><span data-stu-id="ab52a-138">However, provisioning that many resources would be very inefficient.</span></span> <span data-ttu-id="ab52a-139">大多数情况下，应用程序并不需要这么大的容量，因此存在闲置的核心，造成资金浪费。</span><span class="sxs-lookup"><span data-stu-id="ab52a-139">Most of the time, the application will not need that much capacity, so there would be idle cores, costing money without adding value.</span></span>

<span data-ttu-id="ab52a-140">更好的做法是将传入请求放入缓冲区，并让充当负载调节器。</span><span class="sxs-lookup"><span data-stu-id="ab52a-140">A better approach is to put the incoming requests into a buffer, and let the buffer act as a load leveler.</span></span> <span data-ttu-id="ab52a-141">如果采用这种设计，引入服务必须能够在短时间内应对最大引入速率，但后端服务只需处理最大持续负载。</span><span class="sxs-lookup"><span data-stu-id="ab52a-141">With this design, the Ingestion service must be able to handle the maximum ingestion rate over short periods, but the backend services only need to handle the maximum sustained load.</span></span> <span data-ttu-id="ab52a-142">如果在前端缓冲请求，则后端服务应该不需要处理较大的流量高峰。</span><span class="sxs-lookup"><span data-stu-id="ab52a-142">By buffering at the front end, the backend services shouldn't need to handle large spikes in traffic.</span></span> <span data-ttu-id="ab52a-143">根据无人机交付应用程序的规模要求，比较适合使用 [Azure 事件中心](/azure/event-hubs/)进行负载调节。</span><span class="sxs-lookup"><span data-stu-id="ab52a-143">At the scale required for the Drone Delivery application, [Azure Event Hubs](/azure/event-hubs/) is a good choice for load leveling.</span></span> <span data-ttu-id="ab52a-144">事件中心提供较低的延迟和较高的吞吐量，是能够处理较高引入量的经济高效解决方案。</span><span class="sxs-lookup"><span data-stu-id="ab52a-144">Event Hubs offers low latency and high throughput, and is a cost effective solution at high ingestion volumes.</span></span> 

<span data-ttu-id="ab52a-145">在测试中，我们使用了标准层事件中心，其中包括 32 个分区和 100 个吞吐量单位。</span><span class="sxs-lookup"><span data-stu-id="ab52a-145">For our testing, we used a Standard tier event hub with 32 partitions and 100 throughput units.</span></span> <span data-ttu-id="ab52a-146">我们已观察到，引入速率大约为 32000 个事件/秒，延迟大约为 90 毫秒。</span><span class="sxs-lookup"><span data-stu-id="ab52a-146">We observed about 32K events / second ingestion, with latency around 90ms.</span></span> <span data-ttu-id="ab52a-147">目前的默认限制为 20 个吞吐量单位，但 Azure 客户可以通过填写支持请求来请求更多的吞吐量单位。</span><span class="sxs-lookup"><span data-stu-id="ab52a-147">Currently the default limit is 20 throughput units, but Azure customers can request additional throughput units by filing a support request.</span></span> <span data-ttu-id="ab52a-148">有关详细信息，请参阅[事件中心配额](/azure/event-hubs/event-hubs-quotas)。</span><span class="sxs-lookup"><span data-stu-id="ab52a-148">See [Event Hubs quotas](/azure/event-hubs/event-hubs-quotas) for more information.</span></span> <span data-ttu-id="ab52a-149">与所有性能指标一样，许多因素会影响性能，例如消息有效负载大小，因此，请不要将这些数字视为基准。</span><span class="sxs-lookup"><span data-stu-id="ab52a-149">As with all performance metrics, many factors can affect performance, such as message payload size, so don't interpret these numbers as a benchmark.</span></span> <span data-ttu-id="ab52a-150">如果需要更大的吞吐量，引入服务可以在多个事件中心之间分片。</span><span class="sxs-lookup"><span data-stu-id="ab52a-150">If more throughput is needed, the Ingestion service can shard across more than one event hub.</span></span> <span data-ttu-id="ab52a-151">要进一步提高吞吐率，[事件中心专用版](/azure/event-hubs/event-hubs-dedicated-overview)可提供单租户部署，此类部署每秒可以引入 200 万以上的事件。</span><span class="sxs-lookup"><span data-stu-id="ab52a-151">For even higher throughput rates, [Event Hubs Dedicated](/azure/event-hubs/event-hubs-dedicated-overview) offers single-tenant deployments that can ingress over 2 million events per second.</span></span>

<span data-ttu-id="ab52a-152">必须了解事件中心如何实现这么高的吞吐量，因为这会影响到客户端使用事件中心内的消息的方式。</span><span class="sxs-lookup"><span data-stu-id="ab52a-152">It's important to understand how Event Hubs can achieve such high throughput, because that affects how a client should consume messages from Event Hubs.</span></span> <span data-ttu-id="ab52a-153">事件中心不实施队列，</span><span class="sxs-lookup"><span data-stu-id="ab52a-153">Event Hubs does not implement a *queue*.</span></span> <span data-ttu-id="ab52a-154">而是实施事件流。</span><span class="sxs-lookup"><span data-stu-id="ab52a-154">Rather, it implements an *event stream*.</span></span> 

<span data-ttu-id="ab52a-155">使用队列时，单个使用者可能会从队列中删除消息，而下一个使用者将看不到该消息。</span><span class="sxs-lookup"><span data-stu-id="ab52a-155">With a queue, an individual consumer can remove a message from the queue, and the next consumer won't see that message.</span></span> <span data-ttu-id="ab52a-156">因此，借助队列可以使用[使用者竞争模式](../patterns/competing-consumers.md)来并行处理消息和提高可伸缩性。</span><span class="sxs-lookup"><span data-stu-id="ab52a-156">Queues therefore allow you to use a [Competing Consumers pattern](../patterns/competing-consumers.md) to process messages in parallel and improve scalability.</span></span> <span data-ttu-id="ab52a-157">为了提高弹性，使用者可在消息上保留一把锁，并在处理完该消息后释放该锁。</span><span class="sxs-lookup"><span data-stu-id="ab52a-157">For greater resiliency, the consumer holds a lock on the message and releases the lock when it's done processing the message.</span></span> <span data-ttu-id="ab52a-158">如果使用者发生故障 &mdash; 例如，它运行所在的节点崩溃 &mdash; 该锁将会超时，而消息将返回到队列中。</span><span class="sxs-lookup"><span data-stu-id="ab52a-158">If the consumer fails &mdash; for example, the node it's running on crashes &mdash; the lock times out and the message goes back onto the queue.</span></span>

<span data-ttu-id="ab52a-159">在另一方面，事件中心使用流语义。</span><span class="sxs-lookup"><span data-stu-id="ab52a-159">Event Hubs, on the other hand, uses streaming semantics.</span></span> <span data-ttu-id="ab52a-160">使用者根据自身的步调独立读取流。</span><span class="sxs-lookup"><span data-stu-id="ab52a-160">Consumers read the stream independently at their own pace.</span></span> <span data-ttu-id="ab52a-161">每个使用者负责跟踪它当前在流中的位置。</span><span class="sxs-lookup"><span data-stu-id="ab52a-161">Each consumer is responsible for keeping track of its current position in the stream.</span></span> <span data-ttu-id="ab52a-162">使用者应该根据某种预定义的间隔将其当前位置写入持久性存储。</span><span class="sxs-lookup"><span data-stu-id="ab52a-162">A consumer should write its current position to persistent storage at some predefined interval.</span></span> <span data-ttu-id="ab52a-163">这样，如果使用者遇到故障（例如，使用者崩溃，或主机故障），则新实例可以继续从上一个记录的位置读取流。</span><span class="sxs-lookup"><span data-stu-id="ab52a-163">That way, if the consumer experiences a fault (for example, the consumer crashes, or the host fails), then a new instance can resume reading the stream from the last recorded position.</span></span> <span data-ttu-id="ab52a-164">此过程称为“检查点设置”。</span><span class="sxs-lookup"><span data-stu-id="ab52a-164">This process is called *checkpointing*.</span></span> 

<span data-ttu-id="ab52a-165">出于性能原因，使用者通常不会在每条消息的后面设置检查点，</span><span class="sxs-lookup"><span data-stu-id="ab52a-165">For performance reasons, a consumer generally doesn't checkpoint after each message.</span></span> <span data-ttu-id="ab52a-166">而是根据某个固定的间隔设置检查点，例如，在处理 *n* 条消息之后或者每隔 *n* 秒设置检查点。</span><span class="sxs-lookup"><span data-stu-id="ab52a-166">Instead, it checkpoints at some fixed interval, for example after processing *n* messages, or every *n* seconds.</span></span> <span data-ttu-id="ab52a-167">因此，如果某个使用者发生故障，则某些事件可能会处理两次，因为新实例始终从最后一个检查点拾取消息。</span><span class="sxs-lookup"><span data-stu-id="ab52a-167">As a consequence, if a consumer fails, some events may get processed twice, because a new instance always picks up from the last checkpoint.</span></span> <span data-ttu-id="ab52a-168">利弊：密集的检查点可能会降低性能，但设置稀疏的检查点则意味着在发生故障后需要重放更多事件。</span><span class="sxs-lookup"><span data-stu-id="ab52a-168">There is a tradeoff: Frequent checkpoints can hurt performance, but sparse checkpoints mean you will replay more events after a failure.</span></span>

<span data-ttu-id="ab52a-169">事件中心不是针对竞争性使用者设计的。</span><span class="sxs-lookup"><span data-stu-id="ab52a-169">Event Hubs is not designed for competing consumers.</span></span> <span data-ttu-id="ab52a-170">尽管多个使用者可以读取某个流，但每个使用者需要独立遍历该流。</span><span class="sxs-lookup"><span data-stu-id="ab52a-170">Although multiple consumers can read a stream, each traverses the stream independently.</span></span> <span data-ttu-id="ab52a-171">事件中心使用分区使用者模式。</span><span class="sxs-lookup"><span data-stu-id="ab52a-171">Instead, Event Hubs uses a partitioned consumer pattern.</span></span> <span data-ttu-id="ab52a-172">一个事件中心最多包含 32 个分区。</span><span class="sxs-lookup"><span data-stu-id="ab52a-172">An event hub has up to 32 partitions.</span></span> <span data-ttu-id="ab52a-173">可以通过将一个独立的使用者分配到每个分区进行横向缩放。</span><span class="sxs-lookup"><span data-stu-id="ab52a-173">Horizontal scale is achieved by assigning a separate consumer to each partition.</span></span>

<span data-ttu-id="ab52a-174">对于无人机交付工作流而言，这意味着什么？</span><span class="sxs-lookup"><span data-stu-id="ab52a-174">What does this mean for the drone delivery workflow?</span></span> <span data-ttu-id="ab52a-175">为了充分利用事件中心的优势，交付计划程序不能等到处理完每条消息之后才转移到下一条消息。</span><span class="sxs-lookup"><span data-stu-id="ab52a-175">To get the full benefit of Event Hubs, the Delivery Scheduler cannot wait for each message to be processed before moving onto the next.</span></span> <span data-ttu-id="ab52a-176">否则，它要将大部分时间花费在等待网络调用完成上。</span><span class="sxs-lookup"><span data-stu-id="ab52a-176">If it does that, it will spend most of its time waiting for network calls to complete.</span></span> <span data-ttu-id="ab52a-177">相反，它需要使用后端服务的异步调用，来并行处理消息批。</span><span class="sxs-lookup"><span data-stu-id="ab52a-177">Instead, it needs to process batches of messages in parallel, using asynchronous calls to the backend services.</span></span> <span data-ttu-id="ab52a-178">我们知道，选择适当的检查点策略也很重要。</span><span class="sxs-lookup"><span data-stu-id="ab52a-178">As we'll see, choosing the right checkpointing strategy is also important.</span></span>

### <a name="iothub-react"></a><span data-ttu-id="ab52a-201">IotHub React</span><span class="sxs-lookup"><span data-stu-id="ab52a-201">IotHub React</span></span>

<span data-ttu-id="ab52a-208">Akka Streams 也是一个用于从事件中心流式处理事件的十分自然的编程模型。</span><span class="sxs-lookup"><span data-stu-id="ab52a-208">Akka Streams is also a very natural programming model for streaming events from Event Hubs.</span></span> <span data-ttu-id="ab52a-209">我们无需循环访问一批事件，而可以定义一组要应用到每个事件的操作，然后让 Akka Streams 来处理流。</span><span class="sxs-lookup"><span data-stu-id="ab52a-209">Instead of looping through a batch of events, you define a set of operations that will be applied to each event, and let Akka Streams handle the streaming.</span></span> <span data-ttu-id="ab52a-210">Akka Streams 在“源”、“流”和“接收器”方面定义流式处理管道。</span><span class="sxs-lookup"><span data-stu-id="ab52a-210">Akka Streams defines a streaming pipeline in terms of *Sources*, *Flows*, and *Sinks*.</span></span> <span data-ttu-id="ab52a-211">源生成输出流，流处理输入流并生成输出流，接收器使用流且不生成任何输出。</span><span class="sxs-lookup"><span data-stu-id="ab52a-211">A source generates an output stream, a flow processes an input stream and produces an output stream, and a sink consumes a stream without producing any output.</span></span>

<span data-ttu-id="ab52a-212">下面是计划程序服务中用于设置 Akka Streams 管道的代码：</span><span class="sxs-lookup"><span data-stu-id="ab52a-212">Here is the code in the Scheduler service that sets up the Akka Streams pipeline:</span></span>

            }

    });
}

## <a name="idempotent-vs-non-idempotent-operations"></a><span data-ttu-id="ab52a-280">幂等与非幂等操作</span><span class="sxs-lookup"><span data-stu-id="ab52a-280">Idempotent vs non-idempotent operations</span></span>

## <a name="example-idempotent-operations"></a><span data-ttu-id="ab52a-303">示例：幂等操作</span><span class="sxs-lookup"><span data-stu-id="ab52a-303">Example: Idempotent operations</span></span>

<span data-ttu-id="ab52a-304">HTTP 规范中规定，PUT 方法必须是幂等的。</span><span class="sxs-lookup"><span data-stu-id="ab52a-304">The HTTP specification states that PUT methods must be idempotent.</span></span> <span data-ttu-id="ab52a-305">该规范对幂等的定义如下：</span><span class="sxs-lookup"><span data-stu-id="ab52a-305">The specification defines idempotent this way:</span></span>

>  <span data-ttu-id="ab52a-306">如果使用某个请求方法的多个相同请求对服务器造成的预期影响与单个此类请求所造成的影响相同，则认为该方法是“幂等的”。</span><span class="sxs-lookup"><span data-stu-id="ab52a-306">A request method is considered "idempotent" if the intended effect on the server of multiple identical requests with that method is the   same as the effect for a single such request.</span></span> <span data-ttu-id="ab52a-307">([RFC 7231](https://tools.ietf.org/html/rfc7231#section-4))</span><span class="sxs-lookup"><span data-stu-id="ab52a-307">([RFC 7231](https://tools.ietf.org/html/rfc7231#section-4))</span></span>

<span data-ttu-id="ab52a-308">创建新实体时，必须了解 PUT 与 POST 语义之间的区别。</span><span class="sxs-lookup"><span data-stu-id="ab52a-308">It's important to understand the difference between PUT and POST semantics when creating a new entity.</span></span> <span data-ttu-id="ab52a-309">在这两种情况下，客户端都会在请求正文中发送实体的表示形式。</span><span class="sxs-lookup"><span data-stu-id="ab52a-309">In both cases, the client sends a representation of an entity in the request body.</span></span> <span data-ttu-id="ab52a-310">但 URI 的含义有所不同。</span><span class="sxs-lookup"><span data-stu-id="ab52a-310">But the meaning of the URI is different.</span></span>

- <span data-ttu-id="ab52a-311">对于 POST 方法，URI 表示新实体的父资源，例如集合。</span><span class="sxs-lookup"><span data-stu-id="ab52a-311">For a POST method, the URI represents a parent resource of the new entity, such as a collection.</span></span> <span data-ttu-id="ab52a-312">例如，若要创建新的交付项，URI 可能是 `/api/deliveries`。</span><span class="sxs-lookup"><span data-stu-id="ab52a-312">For example, to create a new delivery, the URI might be `/api/deliveries`.</span></span> <span data-ttu-id="ab52a-313">服务器将创建实体并为其分配新 URI，例如 `/api/deliveries/39660`。</span><span class="sxs-lookup"><span data-stu-id="ab52a-313">The server creates the entity and assigns it a new URI, such as `/api/deliveries/39660`.</span></span> <span data-ttu-id="ab52a-314">此 URI 将在响应的 Location 标头中返回。</span><span class="sxs-lookup"><span data-stu-id="ab52a-314">This URI is returned in the Location header of the response.</span></span> <span data-ttu-id="ab52a-315">每当客户端发送请求时，服务器都会创建一个具有新 URI 的新实体。</span><span class="sxs-lookup"><span data-stu-id="ab52a-315">Each time the client sends a request, the server will create a new entity with a new URI.</span></span>

<span data-ttu-id="ab52a-322">下面交付服务的 PUT 方法实现。</span><span class="sxs-lookup"><span data-stu-id="ab52a-322">Here is the Delivery service's implementation of the PUT method.</span></span> 

```csharp
[HttpPut("{id}")]
[ProducesResponseType(typeof(Delivery), 201)]
[ProducesResponseType(typeof(void), 204)]
public async Task<IActionResult> Put([FromBody]Delivery delivery, string id)
{
    logger.LogInformation("In Put action with delivery {Id}: {@DeliveryInfo}", id, delivery.ToLogInfo());
    try
    {
        var internalDelivery = delivery.ToInternal();

        // Create the new delivery entity.
        await deliveryRepository.CreateAsync(internalDelivery);

        // Create a delivery status event.
        var deliveryStatusEvent = new DeliveryStatusEvent { DeliveryId = delivery.Id, Stage = DeliveryEventType.Created };
        await deliveryStatusEventRepository.AddAsync(deliveryStatusEvent);

        // Return HTTP 201 (Created)
        return CreatedAtRoute("GetDelivery", new { id= delivery.Id }, delivery);
    }
    catch (DuplicateResourceException)
    {
        // This method is mainly used to create deliveries. If the delivery already exists then update it.
        logger.LogInformation("Updating resource with delivery id: {DeliveryId}", id);

        var internalDelivery = delivery.ToInternal();
        await deliveryRepository.UpdateAsync(id, internalDelivery);

        // Return HTTP 204 (No Content)
        return NoContent();
    }
}
```

<span data-ttu-id="ab52a-323">大多数请求预期会创建新实体，因此，该方法将对存储库对象乐观调用 `CreateAsync`，然后通过更新资源来处理任何重复资源异常。</span><span class="sxs-lookup"><span data-stu-id="ab52a-323">It's expected that most requests will create a new entity, so the method optimistically calls `CreateAsync` on the repository object, and then handles any duplicate-resource exceptions by updating the resource instead.</span></span>
