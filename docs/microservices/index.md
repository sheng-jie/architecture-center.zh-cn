## <a name="why-build-microservices"></a><span data-ttu-id="e47eb-118">为何要构建微服务？</span><span class="sxs-lookup"><span data-stu-id="e47eb-118">Why build microservices?</span></span>

<span data-ttu-id="e47eb-119">在微服务体系结构中，应用程序由小型的独立服务构成。</span><span class="sxs-lookup"><span data-stu-id="e47eb-119">In a microservices architecture, the application is composed of small, independent services.</span></span> <span data-ttu-id="e47eb-120">下面是微服务的一些典型特征：</span><span class="sxs-lookup"><span data-stu-id="e47eb-120">Here are some of the defining characteristics of microservices:</span></span>

- <span data-ttu-id="e47eb-121">每个微服务实现单个业务功能。</span><span class="sxs-lookup"><span data-stu-id="e47eb-121">Each microservice implements a single business capability.</span></span>
- <span data-ttu-id="e47eb-122">微服务很小，一个小规模的开发人员团队就能编写和维护它。</span><span class="sxs-lookup"><span data-stu-id="e47eb-122">A microservice is small enough that a single small team of developers can write and maintain it.</span></span>
- <span data-ttu-id="e47eb-123">微服务在单独的进程中运行，通过妥善定义的 API 或消息传递模式进行通信。</span><span class="sxs-lookup"><span data-stu-id="e47eb-123">Microservices run in separate processes, communicating through well-defined APIs or messaging patterns.</span></span> 
- <span data-ttu-id="e47eb-124">微服务不共享数据存储或数据架构。</span><span class="sxs-lookup"><span data-stu-id="e47eb-124">Microservices do not share data stores or data schemas.</span></span> <span data-ttu-id="e47eb-125">每个微服务负责管理自身的数据。</span><span class="sxs-lookup"><span data-stu-id="e47eb-125">Each microservice is responsible for managing its own data.</span></span> 
- <span data-ttu-id="e47eb-126">微服务有独立的代码库，且不共享源代码。</span><span class="sxs-lookup"><span data-stu-id="e47eb-126">Microservices have separate code bases, and do not share source code.</span></span> <span data-ttu-id="e47eb-127">不过，它们共用实用工具库。</span><span class="sxs-lookup"><span data-stu-id="e47eb-127">They may use common utility libraries, however.</span></span>
- <span data-ttu-id="e47eb-128">可以独立部署和更新每个微服务。</span><span class="sxs-lookup"><span data-stu-id="e47eb-128">Each microservice can be deployed and updated independently of other services.</span></span> 

<span data-ttu-id="e47eb-129">如果一切正常，则微服务可以提供诸多有利的优势：</span><span class="sxs-lookup"><span data-stu-id="e47eb-129">Done correctly, microservices can provide a number of useful benefits:</span></span>

- <span data-ttu-id="e47eb-130">**敏捷性。**</span><span class="sxs-lookup"><span data-stu-id="e47eb-130">**Agility.**</span></span> <span data-ttu-id="e47eb-131">由于微服务是独立部署的，因此我们可以更轻松地管理 bug 修复和功能发布。</span><span class="sxs-lookup"><span data-stu-id="e47eb-131">Because microservices are deployed independently, it's easier to manage bug fixes and feature releases.</span></span> <span data-ttu-id="e47eb-132">无需重新部署整个应用程序即可更新服务，出现问题时可回滚更新。</span><span class="sxs-lookup"><span data-stu-id="e47eb-132">You can update a service without redeploying the entire application, and roll back an update if something goes wrong.</span></span> <span data-ttu-id="e47eb-133">在许多传统应用程序中，如果在应用程序的某个部分发现 bug，则整个发布过程可能会停滞；因此，只能在等到修复 bug 之后，才能集成、测试和发布新功能。</span><span class="sxs-lookup"><span data-stu-id="e47eb-133">In many traditional applications, if a bug is found in one part of the application, it can block the entire release process; as a result, new features may be held up waiting for a bug fix to be integrated, tested, and published.</span></span>  

- <span data-ttu-id="e47eb-134">**小代码，小团队。**</span><span class="sxs-lookup"><span data-stu-id="e47eb-134">**Small code, small teams.**</span></span> <span data-ttu-id="e47eb-135">微服务应该足够小，单个功能团队就能构建、测试和部署。</span><span class="sxs-lookup"><span data-stu-id="e47eb-135">A microservice should be small enough that a single feature team can build, test, and data-ttu-id="e47eb-137">在大型整体应用程序中，经过一段时间后，代码依赖项往往会变得混杂，因此，添加新功能需要在多个位置改动代码。</span><span class="sxs-lookup"><span data-stu-id="e47eb-137">In a large monolithic application, there is a tendency over time for code dependencies to become tangled, so that adding a new feature requires touching code in a lot of places.</span></span> <span data-ttu-id="e47eb-138">如果不共享代码或数据存储，则微服务体系结构可将依赖项减到最少，使新功能的添加变得更容易。</span><span class="sxs-lookup"><span data-stu-id="e47eb-138">By not sharing code or data stores, a microservices architecture class="sxs-lookup"><span data-stu-id="e47eb-142">But the point is that large groups tend be less productive, because communication is slower, management overhead goes up, and agility - <span data-ttu-id="e47eb-143">**混合技术**。</span><span class="sxs-lookup"><span data-stu-id="e47eb-143">**Mix of technologies**.</span></span> <span data-ttu-id="e47eb-144">团队可以选择最适合其服务的技术，并适当地使用混合技术堆栈。</span><span class="sxs-lookup"><span data-stu-id="e47eb-144">Teams can pick the technology that best fits their service, using a mix of technology stacks as appropriate.</span></span> 

- <span data-ttu-id="e47eb-145">**复原能力**。</span><span class="sxs-lookup"><span data-stu-id="e47eb-145">**Resiliency**.</span></span> <span data-ttu-id="e47eb-146">单个微服务在不可用的情况下不会中断整个应用程序，但前提是所有上游微服务能够正确处理故障（例如，通过实施断路机制）。</span><span class="sxs-lookup"><span data-stu-id="e47eb-146">If an individual microservice becomes unavailable, it won't disrupt the entire application, as long as any upstream microservices are designed to handle faults correctly (for example, by implementing circuit breaking).</span></span>

- <span data-ttu-id="e47eb-147">**可伸缩性**。</span><span class="sxs-lookup"><span data-stu-id="e47eb-147">**Scalability**.</span></span> <span data-ttu-id="e47eb-148">微服务体系结构允许每个微服务独立进行缩放。</span><span class="sxs-lookup"><span data-stu-id="e47eb-148">A microservices architecture allows each microservice to be scaled independently of the others.</span></span> <span data-ttu-id="e47eb-149">这样，我们便可以横向扩展需要更多资源的子系统，而无需横向扩展整个应用程序。</span><span class="sxs-lookup"><span data-stu-id="e47eb-149">That lets you scale out subsystems that require more resources, without scaling out the entire application.</span></span> <span data-ttu-id="e47eb-150">如果将服务部署在容器中，则还可以将更高密度的微服务打包到一台主机中，从而提高资源的利用效率。</span><span class="sxs-lookup"><span data-stu-id="e47eb-150">If you deploy services inside containers, you can also pack a higher density of microservices onto a single host, which allows for more efficient utilization of resources.</span></span>

- <span data-ttu-id="e47eb-151">**数据隔离**。</span><span class="sxs-lookup"><span data-stu-id="e47eb-151">**Data isolation**.</span></span> <span data-ttu-id="e47eb-152">执行架构更新要容易得多，因为只会影响单个微服务。</span><span class="sxs-lookup"><span data-stu-id="e47eb-152">It is much easier to perform schema updates, because only a single microservice is impacted.</span></span> <span data-ttu-id="e47eb-153">在整体应用程序中，架构更新可能极具挑战性，因为应用程序的不同部分可能都要取用相同的数据，对架构进行任何更改都会带来风险。</span><span class="sxs-lookup"><span data-stu-id="e47eb-153">In a monolithic application, schema updates can become very challenging, because different parts of the application may all touch the same data, making any alterations to the schema risky.</span></span>
